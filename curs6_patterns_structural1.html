Design Patterns Structurale — Partea I

Pattern-urile structurale descriu cum se compun clasele și obiectele pentru a forma structuri mai mari și mai flexibile. Astăzi acoperim trei din cele șapte: Adapter — face compatibile interfețe incompatibile; Decorator — adaugă responsabilități fără moștenire; Facade — simplifică accesul la un subsistem complex.

Obiective
  • Înțelegeți rolul pattern-urilor structurale și cum diferă de cele creaționale
  • Aplicați Adapter pentru a integra componente cu interfețe incompatibile
  • Distingeți Object Adapter (compoziție) de Class Adapter (moștenire) și știți când să le alegeți
  • Folosiți Decorator pentru a extinde comportamentul la runtime fără subclasare
  • Implementați Facade pentru a reduce cuplarea dintre client și un subsistem complex

01Pattern-uri structurale — context

Pattern-urile creaționale răspundeau la întrebarea „cum creăm obiectele?". Pattern-urile structurale răspund la o altă întrebare: „cum compunem clasele și obiectele în structuri mai mari, fără să pierdem flexibilitatea?"

GoF definește șapte pattern-uri structurale. Le parcurgem în două cursuri:

PatternProblema rezolvatăCurs
AdapterInterfețe incompatibile — face două componente să colaboreze6 — astăzi
DecoratorAdaugă responsabilități unui obiect dinamic, fără moștenire6 — astăzi
FacadeSimplifică accesul la un subsistem complex6 — astăzi
CompositeTratează uniform obiecte individuale și colecții arborescente7
ProxyControlează accesul la un obiect prin intermediar7
BridgeSepară abstracția de implementare — variază independent7
FlyweightPartajează starea comună între multe obiecte mici7

02Adapter

🔌
Adapter
Convertește interfața unei clase în interfața pe care clienții o așteaptă. Permite colaborarea claselor cu interfețe incompatibile.
Structural GoF

Scenariul apare frecvent la integrarea de biblioteci externe sau sisteme legacy: aveți o componentă care funcționează corect, dar cu o interfață diferită de cea pe care o așteaptă sistemul vostru. Rescrierea componentei nu este fezabilă — Adapter o integrează fără să o modifice.

Structura generică (GoF)

Diagramă de clase — structura statică
classDiagram class Target { <<interface>> +request() } class Adapter { -adaptee Adaptee +request() } class Adaptee { +specificRequest() } class Client Target <|.. Adapter Adapter --> Adaptee : wraps Client ..> Target : uses
Diagramă de secvență — interacțiunea obiectelor
sequenceDiagram participant C as Client participant A as Adapter participant Ad as Adaptee C->>A: request() A->>Ad: specificRequest() Ad-->>A: rezultat brut A-->>C: rezultat adaptat

Structura C# — participanți canonici

Adapter — structura generică C# 
// ── Target ───────────────────────────────────────────────────────────
// Interfața pe care clientul o cunoaște și o folosește
public interface ITarget
{
    string Request();
}

// ── Adaptee ───────────────────────────────────────────────────────────
// Clasa existentă cu interfața incompatibilă — nu o modificăm
public class Adaptee
{
    public string SpecificRequest()
        => "rezultat din Adaptee";
}

// ── Object Adapter ────────────────────────────────────────────────────
// Implementează Target, conține Adaptee prin compoziție (varianta preferată)
public class Adapter : ITarget
{
    private readonly Adaptee _adaptee;

    public Adapter(Adaptee adaptee) => _adaptee = adaptee;

    public string Request()
    {
        // Traduce apelul din interfața Target în apelul specific Adaptee
        var result = _adaptee.SpecificRequest();
        return $"Adapter: [{result}]";
    }
}

// ── Client ────────────────────────────────────────────────────────────
// Depinde exclusiv de ITarget — nu știe că în spate există Adaptee
public class Client
{
    private readonly ITarget _target;

    public Client(ITarget target) => _target = target;

    public void Run() => Console.WriteLine(_target.Request());
}

// ── Asamblare ─────────────────────────────────────────────────────────
var adaptee = new Adaptee();
var adapter = new Adapter(adaptee);
var client  = new Client(adapter);
client.Run(); // Output: Adapter: [rezultat din Adaptee]

Object Adapter vs Class Adapter

Există două variante ale pattern-ului. Object Adapter folosește compoziția — este varianta preferată, funcționează și cu subclasele lui Adaptee și respectă principiul „favorizează compoziția față de moștenire". Class Adapter folosește moștenirea — în C# nu este posibilă moștenirea multiplă din clase concrete, deci este aplicabil doar când Adaptee este o interfață.

❌ Fără Adapter — interfețe incompatibile
incompatibilitate directă C# 
// Interfața așteptată de sistemul nostru
public interface IPaymentProcessor
{
    bool ProcessPayment(decimal amount);
}

// Biblioteca externă — nu putem modifica
public class PayPalGateway
{
    public PayPalResponse MakePayment(
        double amount, string currency) { ... }
}

// ❌ PayPalGateway nu implementează IPaymentProcessor:
// - type mismatch: decimal vs double
// - signature diferită: parametru currency în plus
// - return type diferit: PayPalResponse vs bool
✅ Object Adapter — compatibilitate prin compoziție
PayPalAdapter.cs C# 
public class PayPalAdapter : IPaymentProcessor
{
    private readonly PayPalGateway _gateway;

    public PayPalAdapter(PayPalGateway gateway)
        => _gateway = gateway;

    public bool ProcessPayment(decimal amount)
    {
        // decimal → double + adăugare currency
        var response = _gateway.MakePayment(
            (double)amount, "EUR");
        // PayPalResponse → bool
        return response.StatusCode == "SUCCESS";
    }
}

// OrderService nu știe că în spate e PayPal
IPaymentProcessor processor =
    new PayPalAdapter(new PayPalGateway());
orderService.PlaceOrder(cart, processor);
🔗
Relație cu DIP și OCP

OrderService depinde de IPaymentProcessor — respectă DIP. Adăugarea unui nou furnizor de plăți (Stripe, Braintree) înseamnă un nou Adapter, fără modificarea OrderService — respectă OCP. Adapter este mecanismul prin care DIP devine practic în contexte de integrare.

✓ Când se folosește
Când trebuie să integrați o clasă existentă cu o interfață incompatibilă și nu puteți modifica sursa. Când reutilizați cod legacy sau biblioteci externe fără control asupra codului sursă.
✗ Când nu se potrivește
Când diferența dintre interfețe este atât de mare încât adaptarea necesită business logic complex — atunci e nevoie de un Anti-Corruption Layer, nu de un simplu Adapter.

03Decorator

🎁
Decorator
Adaugă responsabilități suplimentare unui obiect în mod dinamic. Decoratorii sunt o alternativă flexibilă la moștenire pentru extinderea funcționalității.
Structural GoF

Moștenirea extinde funcționalitatea la compile-time și produce un număr exponențial de subclase. Dacă aveți un cititor de date și vreți să adăugați opțional criptare, compresie și logging — prin moștenire aveți nevoie de 7 subclase pentru toate combinațiile posibile. Decorator rezolvă asta prin wrapping la runtime.

Structura generică (GoF)

Diagramă de clase — structura statică
classDiagram class Component { <<interface>> +operation() } class ConcreteComponent { +operation() } class Decorator { <<abstract>> -component Component +operation() } class ConcreteDecoratorA { +operation() +addedBehavior() } class ConcreteDecoratorB { +operation() } Component <|.. ConcreteComponent Component <|.. Decorator Decorator --> Component : wraps Decorator <|-- ConcreteDecoratorA Decorator <|-- ConcreteDecoratorB
Diagramă de secvență — interacțiunea obiectelor
sequenceDiagram participant C as Client participant DA as DecoratorA participant DB as DecoratorB participant CC as ConcreteComponent C->>DA: operation() DA->>DB: operation() DB->>CC: operation() CC-->>DB: rezultat de baza DB-->>DA: rezultat + comportament B DA-->>C: rezultat + comportament A si B

Structura C# — participanți canonici

Decorator — structura generică C# 
// ── Component ─────────────────────────────────────────────────────────
// Interfața comună pentru ConcreteComponent și toți decoratorii
public interface IComponent
{
    string Operation();
}

// ── ConcreteComponent ─────────────────────────────────────────────────
// Obiectul de bază care face munca reală
public class ConcreteComponent : IComponent
{
    public string Operation() => "ConcreteComponent";
}

// ── Decorator (abstract) ──────────────────────────────────────────────
// Menține referința la IComponent și implementează aceeași interfață
public abstract class Decorator : IComponent
{
    protected readonly IComponent _inner;

    protected Decorator(IComponent inner) => _inner = inner;

    // Implicit delega la inner — subclasele suprascriu și adaugă comportament
    public virtual string Operation() => _inner.Operation();
}

// ── ConcreteDecoratorA ────────────────────────────────────────────────
public class ConcreteDecoratorA : Decorator
{
    public ConcreteDecoratorA(IComponent inner) : base(inner) {}

    public override string Operation()
    {
        var result = base.Operation(); // apelează inner
        return $"DecoratorA({result})";  // adaugă comportament
    }
}

// ── ConcreteDecoratorB ────────────────────────────────────────────────
public class ConcreteDecoratorB : Decorator
{
    public ConcreteDecoratorB(IComponent inner) : base(inner) {}

    public override string Operation()
    {
        var result = base.Operation();
        return $"DecoratorB({result})";
    }
}

// ── Asamblare — ordinea învelirilor definește ordinea execuției ────────
IComponent component =
    new ConcreteDecoratorA(
        new ConcreteDecoratorB(
            new ConcreteComponent()));

Console.WriteLine(component.Operation());
// Output: DecoratorA(DecoratorB(ConcreteComponent))

Exemplu — cititor de date cu decoratori

IDataReader.cs + decoratori C# 
// Interfața comună — Component
public interface IDataReader
{
    string Read();
}

// Implementare de bază — ConcreteComponent
public class FileDataReader : IDataReader
{
    private readonly string _path;
    public FileDataReader(string path) => _path = path;
    public string Read() => File.ReadAllText(_path);
}

// Decorator de bază — abstract, menține referința la component
public abstract class DataReaderDecorator : IDataReader
{
    protected readonly IDataReader _inner;
    protected DataReaderDecorator(IDataReader inner) => _inner = inner;
    public abstract string Read();
}

// Decorator 1: decriptează după citire
public class EncryptedDataReader : DataReaderDecorator
{
    public EncryptedDataReader(IDataReader inner) : base(inner) {}
    public override string Read()
        => Decrypt(_inner.Read());
    private string Decrypt(string data) => /* AES decrypt */ data;
}

// Decorator 2: decomprimă după citire
public class CompressedDataReader : DataReaderDecorator
{
    public CompressedDataReader(IDataReader inner) : base(inner) {}
    public override string Read()
        => Decompress(_inner.Read());
    private string Decompress(string d) => /* GZip decompress */ d;
}

// Combinare la runtime — ordinea contează!
IDataReader reader =
    new EncryptedDataReader(             // 3. decriptează
        new CompressedDataReader(          // 2. decomprimă
            new FileDataReader("data.bin")  // 1. citește fișier
        ));

string result = reader.Read(); // traversează întreg lanțul
⚠️
Ordinea decoratorilor contează

Fișierul a fost mai întâi comprimat, apoi criptat la scriere — deci la citire trebuie mai întâi decriptat, apoi decomprimat. Dacă inversați ordinea decoratorilor, obțineți date corupte fără nicio eroare la compile time. Această dependență de ordine este responsabilitatea codului care asamblează lanțul.

✓
Decorator este implementarea canonică a OCP

Adăugarea unui LoggingDataReader nu modifică nicio clasă existentă. Se creează o clasă nouă care extinde DataReaderDecorator și se include în lanț. Codul existent nu știe că există noul decorator.

✓ Când se folosește
Când trebuie să adăugați comportamente unui obiect fără să modificați clasa sa. Când combinațiile de comportamente sunt multiple și variabile la runtime. Când moștenirea ar produce o explozie de subclase.
✗ Când nu se potrivește
Când ordinea decoratorilor este critică și greu de gestionat. Când există un singur comportament suplimentar fix — o subclasă simplă este mai clară. Lanțurile lungi de decoratori pot fi dificil de depanat.

04Facade

🏛
Facade
Oferă o interfață simplificată către un subsistem complex. Facade definește o interfață de nivel înalt care face subsistemul mai ușor de utilizat.
Structural GoF

Un subsistem complex are zeci de clase cu relații complicate între ele. Clientul nu ar trebui să cunoască toate aceste detalii pentru a realiza o operație de nivel înalt. Facade expune operațiile comune printr-o interfață simplă, ascunzând complexitatea internă.

Structura generică (GoF)

Diagramă de clase — structura statică
classDiagram class Facade { +operationA() +operationB() } class SubsystemA { +featureA1() +featureA2() } class SubsystemB { +featureB1() } class SubsystemC { +featureC1() } class Client Client ..> Facade : uses Facade --> SubsystemA Facade --> SubsystemB Facade --> SubsystemC
Diagramă de secvență — interacțiunea obiectelor
sequenceDiagram participant C as Client participant F as OrderFacade participant I as InventoryService participant P as PaymentService participant S as ShippingService participant N as NotificationService C->>F: PlaceOrder(cart, payment) F->>I: Reserve(cart) I-->>F: ok F->>P: Charge(payment) P-->>F: transactionId F->>S: Schedule(cart, address) S-->>F: trackingCode F->>N: SendConfirmation(email) F-->>C: OrderResult

Structura C# — participanți canonici

Facade — structura generică C# 
// ── Subsisteme ────────────────────────────────────────────────────────
// Clase complexe cu logică internă — clientul nu le cunoaște direct
public class SubsystemA
{
    public string OperationA1() => "SubsystemA: OperationA1";
    public string OperationA2() => "SubsystemA: OperationA2";
}

public class SubsystemB
{
    public string OperationB1() => "SubsystemB: OperationB1";
}

public class SubsystemC
{
    public string OperationC1() => "SubsystemC: OperationC1";
    public string OperationC2() => "SubsystemC: OperationC2";
}

// ── Facade ────────────────────────────────────────────────────────────
// Interfață simplificată — orchestrează subsistemele, expune operații de nivel înalt
public class Facade
{
    private readonly SubsystemA _a;
    private readonly SubsystemB _b;
    private readonly SubsystemC _c;

    public Facade(SubsystemA a, SubsystemB b, SubsystemC c)
        => (_a, _b, _c) = (a, b, c);

    // Operație de nivel înalt — clientul apelează doar aceasta
    public string Operation()
    {
        var results = new List<string>
        {
            _a.OperationA1(),
            _b.OperationB1(),
            _c.OperationC1(),
            _c.OperationC2(),
            _a.OperationA2(),
        };
        return string.Join("
", results);
    }
}

// ── Client ────────────────────────────────────────────────────────────
// Nu cunoaște SubsystemA/B/C — depinde doar de Facade
var facade = new Facade(new SubsystemA(), new SubsystemB(), new SubsystemC());
Console.WriteLine(facade.Operation());

Exemplu — sistem de comenzi online

OrderFacade.cs C# 
// Subsisteme individuale cu responsabilități clare
public class InventoryService
{
    public bool Reserve(Cart cart) => true;
    public void Release(Cart cart) { }
}
public class PaymentService
{
    public string Charge(PaymentInfo info, decimal amount) => "TXN-001";
    public void   Refund(string transactionId) { }
}
public class ShippingService
{
    public string Schedule(Cart cart, Address addr) => "TRACK-001";
}
public class NotificationService
{
    public void SendConfirmation(string email, OrderResult r) { }
}

// ── Facade — orchestrează toate subsistemele ─────────────────────────
public class OrderFacade
{
    private readonly InventoryService    _inventory;
    private readonly PaymentService      _payment;
    private readonly ShippingService     _shipping;
    private readonly NotificationService _notification;

    public OrderFacade(InventoryService inv, PaymentService pay,
                       ShippingService ship, NotificationService notif)
    { _inventory = inv; _payment = pay; _shipping = ship; _notification = notif; }

    // Operație de nivel înalt — clientul apelează doar aceasta
    public OrderResult PlaceOrder(Cart cart, PaymentInfo payment, Address address)
    {
        if (!_inventory.Reserve(cart))
            throw new InvalidOperationException("Stoc insuficient");

        string transactionId;
        try
        {
            transactionId = _payment.Charge(payment, cart.Total());
        }
        catch
        {
            _inventory.Release(cart); // rollback stoc
            throw;
        }

        var trackingCode = _shipping.Schedule(cart, address);
        var result = new OrderResult(transactionId, trackingCode);
        _notification.SendConfirmation(payment.Email, result);
        return result;
    }
}

// Client — o singură linie, zero cunoaștere a subsistemelor
var result = orderFacade.PlaceOrder(cart, paymentInfo, address);
💡
Facade nu interzice accesul direct

Facade nu blochează accesul la subsisteme. Clienții avansați pot în continuare să folosească InventoryService sau PaymentService direct dacă au nevoie de control granular (fine-grained access). Facade oferă o cale simplă pentru cazul comun — nu o barieră obligatorie.

🧭
Adapter vs Decorator vs Facade

Adapter rezolvă o problemă de incompatibilitate — traduce o interfață în alta. Decorator rezolvă o problemă de extensibilitate — adaugă comportamente la runtime. Facade rezolvă o problemă de complexitate — simplifică accesul la un subsistem. Cele trei răspund la întrebări diferite, chiar dacă structural toate „învelesc" ceva.

✓ Când se folosește
Când un subsistem complex expune prea multe detalii clientului. Când vreți să reduceți cuplarea dintre niveluri arhitecturale. La granița dintre straturi: UI → Service, Service → Infrastructure.
✗ Când nu se potrivește
Când Facade devine un „god object" cu zeci de metode care știe de tot — semnalează că subsistemul nu este bine structurat. Facade nu înlocuiește separarea corectă a responsabilităților.

05Comparație și ghid de alegere

Cele trei pattern-uri par similare la suprafață — toate fac wrap la un obiect sau un set de obiecte. Diferența stă în intenție:

ÎntrebarePattern
Am o componentă cu interfața greșită și nu pot să o modific Adapter
Vreau să adaug comportament unui obiect, combinabil la runtime, fără subclassing Decorator
Am un subsistem complex și vreau să ofer clienților un singur punct de acces simplu Facade
Adapter schimbă interfața unui obiect. Decorator adaugă responsabilități unui obiect. Facade definește o nouă interfață pentru un întreg subsistem. — GoF

06Laborator 6

Proiect independent de la zero care aplică explicit cele trei pattern-uri structurale. Tema: un sistem de procesare a documentelor — parsing, validare, transformare, salvare — cu suport pentru formate multiple și comportamente opționale.

🔬

Sistem de Procesare a Documentelor

Laborator 6 — aplicație .NET cu Adapter, Decorator și Facade

Sistemul procesează documente din surse diferite (fișiere XML legacy, fișiere JSON), le validează, le transformă în format intern și le salvează. Fiecare etapă poate fi decorată cu comportamente suplimentare: logging, caching, validare.

Cerințe
ADAPTER Cerința 1 — Integrarea unui parser legacy
  • Interfața sistemului: IDocumentParser cu metoda Parse(string content): Document
  • Parser extern legacy (nu se poate modifica): LegacyXmlParser cu metoda ParseXml(XmlDocument xml): LegacyDocument
  • Implementați XmlParserAdapter : IDocumentParser care convertește string → XmlDocument, apelează LegacyXmlParser și convertește LegacyDocument → Document
  • Adăugați și un JsonDocumentParser : IDocumentParser nativ — demonstrând că cele două parsere sunt interschimbabile față de restul sistemului
DECORATOR Cerința 2 — Comportamente opționale la parsare
  • Implementați clasa de bază abstractă DocumentParserDecorator : IDocumentParser
  • LoggingDocumentParser — loghează în consolă timpul de start, end și dimensiunea documentului rezultat
  • ValidationDocumentParser — validează documentul după parsare (titlu nenul, conținut minim 10 caractere); aruncă ValidationException dacă nu este valid
  • CachingDocumentParser — cacheează rezultatul pe baza unui hash al conținutului; al doilea apel cu același conținut nu mai declanșează parsing-ul
  • Demonstrați combinarea: new LoggingDocumentParser(new ValidationDocumentParser(new XmlParserAdapter(...)))
FACADE Cerința 3 — Orchestrare prin Facade
  • Implementați DocumentProcessingFacade cu metoda Process(string filePath): ProcessingResult
  • Facade detectează formatul fișierului (XML/JSON), selectează parserul cu decoratorii potriviți, validează, transformă și salvează rezultatul
  • Clientul apelează o singură metodă și primește un ProcessingResult cu status și calea fișierului salvat
  • Erorile sunt tratate în Facade: dacă parsarea sau validarea eșuează, ProcessingResult.IsSuccess = false cu mesajul corespunzător — fără excepție propagată la client
TESTE Cerința 4 — Teste NUnit
  • Test: XmlParserAdapter și JsonDocumentParser returnează același tip Document pentru input echivalent — demonstrează interschimbabilitatea (Adapter)
  • Test: ValidationDocumentParser aruncă ValidationException pentru un document cu titlu gol (Decorator — izolare comportament)
  • Test: CachingDocumentParser apelează parserul interior o singură dată pentru același conținut — al doilea apel vine din cache (Decorator — verificare cu mock)
  • Test: DocumentProcessingFacade.Process() returnează IsSuccess = false pentru conținut invalid, fără excepție (Facade — gestionare erori)
