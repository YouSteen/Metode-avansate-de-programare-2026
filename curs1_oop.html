Recapitulare Programare Orientată pe Obiecte

O privire critică asupra conceptelor OOP — nu ce sunt, ci de ce există și când devin o problemă. Firul roșu al acestui curs: coeziunea și cuplarea.

🎯 Obiective
  • Consolidarea conceptelor OOP cu accent pe motivul existenței lor, nu pe sintaxă
  • Înțelegerea coeziunii și cuplării ca metrici centrale ale calității proiectării
  • Distingerea clară între relațiile dintre clase: asociere, agregare, compoziție, moștenire
  • Identificarea problemelor tipice de proiectare OOP care vor fi rezolvate pe parcursul cursului

01De ce OOP?

Înainte să recapitulăm ce este OOP, merită să înțelegem de ce a apărut. Programarea procedurală funcționează bine la scară mică. Când sistemul crește — sute de funcții, mii de variabile globale, module care se cheamă haotic unele pe altele — mentenanța devine imposibilă.

💡
Problema rezolvată

OOP nu a apărut pentru că cineva a vrut să scrie mai mult cod. A apărut pentru că codul procedural la scară mare nu poate fi menținut — o modificare într-un loc sparge lucruri în 10 alte locuri neașteptate.

OOP propune o soluție: organizarea codului în jurul obiectelor — unități care combină date (stare) cu comportament (metode) și care interacționează prin contracte bine definite. Fiecare obiect are o responsabilitate clară și comunică cu restul sistemului printr-o interfață publică.

Întrebarea fundamentală în OOP — și în tot cursul de față — este:
Cui aparține această responsabilitate?

Această întrebare este mai valoroasă decât orice regulă sintactică. O vei auzi repetat în fiecare curs, de la principiile SOLID până la arhitecturi software.

02Cei 4 Piloni OOP — O Privire Critică

Toți știm definiția manualului. Scopul acestei secțiuni nu este să le redefinim, ci să înțelegem limitele fiecărui pilon și când devin o problemă.

🔒
Încapsularea

Ascunderea stării interne. Obiectul își gestionează singur datele.

→ Nu e doar „câmpuri private"
🧬
Moștenirea

Reutilizarea comportamentului. Specializarea tipurilor.

→ Preferă compoziția (GoF)
🔄

Același apel, comportamente diferite. Fundamentul extensibilității.

→ Interfețele > moștenire
🎭
Abstractizarea

Contracte fără detalii de implementare. Separarea CE de CUM.

→ Abstracte vs. interfețe

Încapsularea

Înțelegerea superficială: „câmpuri private, proprietăți public". Înțelegerea corectă: un obiect își gestionează singur starea internă și nu lasă alte componente să o corupă.

Diferența este subtilă dar esențială. Un câmp privat cu un setter public care acceptă orice valoare nu oferă nicio protecție reală. Încapsularea înseamnă că toate regulile de validare și tranziții de stare sunt controlate de obiect.

❌ Encapsulare falsă — starea este vulnerabilă C# 
public class BankAccount
{
    public decimal Balance { get; set; } // oricine poate seta orice valoare
    public bool IsActive { get; set; }
}

// În altă parte a codului:
account.Balance = -9999;   // nimeni nu te oprește
account.IsActive = false;  // fără nicio verificare
✅ Encapsulare reală — obiectul este responsabil de starea sa C# 
public class BankAccount
{
    public decimal Balance { get; private set; }
    public bool IsActive { get; private set; }

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("Suma trebuie să fie pozitivă");
        if (!IsActive) throw new InvalidOperationException("Cont inactiv");
        Balance += amount;
    }

    public void Withdraw(decimal amount)
    {
        if (amount > Balance) throw new InvalidOperationException("Fonduri insuficiente");
        Balance -= amount;
    }
}
✓
Regula de aur

Dacă starea unui obiect poate fi coruptă din exterior, nu ai încapsulare — ai doar sintaxă privată.

Moștenirea

Moștenirea permite reutilizarea comportamentului și specializarea tipurilor. Este un instrument puternic, dar cel mai adesea abuzat în OOP.

⚠️
Avertisment

Moștenirea profundă (A → B → C → D → E) creează cod fragil: o modificare în clasa de bază poate sparge toate subclasele. Gang of Four spuneau deja în 1994: „Preferă compoziția în fața moștenirii."

Regula practică: folosești moștenirea când relația este cu adevărat IS-A semantic, nu doar structural. Un Square este matematic un Rectangle, dar din perspectiva comportamentului OOP nu este — vom vedea de ce la cursul SOLID (LSP).

Moștenire → Compoziție C# 
// ❌ Moștenire forțată — Employee "este" DatabaseConnection?
public class Employee : DatabaseConnection { ... }

// ✅ Compoziție — Employee "are" un repository
public class Employee
{
    private readonly IEmployeeRepository _repository;

    public Employee(IEmployeeRepository repository)
        => _repository = repository;
}

Polimorfismul

Polimorfismul înseamnă că același apel produce comportamente diferite în funcție de tipul real al obiectului. Este fundamentul extensibilității în OOP.

Există două forme principale în C#: prin moștenire (virtual/override) și prin interfețe. În practica modernă, polimorfismul prin interfețe este preferat — nu creează dependențe de ierarhie și permite unui obiect să implementeze mai multe contracte.

Polimorfism prin interfețe C# 
public interface IShape
{
    double Area();
    string Describe();
}

public class Circle : IShape
{
    private readonly double _radius;
    public Circle(double radius) => _radius = radius;

    public double Area() => Math.PI * _radius * _radius;
    public string Describe() => $"Cerc cu raza {_radius:F2}";
}

public class Rectangle : IShape
{
    private readonly double _w, _h;
    public Rectangle(double w, double h) => (_w, _h) = (w, h);

    public double Area() => _w * _h;
    public string Describe() => $"Dreptunghi {_w}×{_h}";
}

// Clientul lucrează cu IShape — nu știe ce tip concret primește
List<IShape> shapes = new() { new Circle(5), new Rectangle(3, 4) };
double total = shapes.Sum(s => s.Area()); // adaugă orice IShape fără a modifica această linie

Abstractizarea

Abstractizarea înseamnă expunerea unui contract fără detalii de implementare. În C# avem două mecanisme: interfețe și clase abstracte. Nu sunt interschimbabile.

Criteriu Interfață (interface) Clasă abstractă (abstract class)
Scop principal Contract pur — CE face Implementare parțială + contract
Implementare implicită Da (C# 8+, rar recomandat) Da, pentru metode comune
Multiple implementări Da — o clasă poate implementa N interfețe Nu — o clasă moștenește o singură clasă
Stare (câmpuri) Nu Da
Când folosești Contracte pentru comportament. Injecție dependențe. Logică comună + pași variabili (Template Method)
💡
Regula practică

Dacă nu ai logică comună de împărtășit între implementări, folosește interfața. Clasa abstractă este justificată când există comportament comun real care nu aparține niciunei subclase.

03Coeziune & Cuplare — Firul Roșu al Cursului

Dacă ar trebui să alegi două metrici pentru a evalua calitatea unui design OOP, acestea sunt coeziunea și cuplarea. Le vei regăsi implicit în toate principiile SOLID, în design patterns și în arhitecturi software.

Coeziune Mare O clasă / metodă face un singur lucru și îl face bine. Toate elementele ei contribuie la un scop comun. Semnale: numele clasei este precis, metodele lucrează cu aceleași date.
Coeziune Mică Clasa face lucruri neînrudite. Conține metode care nu au nimic în comun. Semnale: nume vagi (Manager, Helper, Util), clase cu 500+ linii.
Cuplare Mică Componentele depind cât mai puțin unele de altele. O schimbare într-un loc nu propagă schimbări în tot sistemul. Dependen­țele sunt prin abstracții (interfețe), nu prin tipuri concrete.
Cuplare Mare Clasele se cunosc direct unele pe altele. new SqlOrderRepository() în interiorul unui serviciu de business creează cuplare concretă. Testarea și înlocuirea componentelor devine imposibilă.
Obiectivul: coeziune mare + cuplare mică.
Acesta este, în esență, scopul tuturor principiilor și pattern-urilor din restul cursului.
Cuplare mare — un serviciu știe prea multe C# 
public class OrderService
{
    public void PlaceOrder(Order order)
    {
        // cuplare concretă: știe că există SQL Server
        var db = new SqlOrderRepository("Server=...");

        // cuplare concretă: știe că există SMTP
        var mailer = new SmtpEmailService("smtp.company.com");

        db.Save(order);
        mailer.SendConfirmation(order);
    }
}
Cuplare mică — serviciul depinde de abstracții C# 
public class OrderService
{
    private readonly IOrderRepository _repository;
    private readonly IEmailService _emailService;

    public OrderService(IOrderRepository repository, IEmailService emailService)
    {
        _repository = repository;
        _emailService = emailService;
    }

    public void PlaceOrder(Order order)
    {
        _repository.Save(order);         // nu știe ce DB
        _emailService.SendConfirmation(order); // nu știe ce provider
    }
}
// Schimbi baza de date sau provider-ul de email? OrderService rămâne neatins.

04Relații între Clase

Un aspect adesea superficial tratat este diferența dintre tipurile de relații. Nu sunt echivalente — exprimă semantici diferite despre ownership și ciclul de viață al obiectelor.

Compoziție → „parte exclusivă" — Order conține OrderLine. Dacă comanda dispare, dispar și liniile. Cicluri de viață comune.
Agregare → „parte partajată" — Team conține Employee. Dacă echipa se desființează, angajații există în continuare.
Asociere → „cunoaște" — Order referențiază Customer. Nu există ownership. Ambele există independent.
Dependență → „folosește temporar" — o metodă primește un parametru de tip ILogger. Relația există doar pe durata apelului.
Moștenire / Realizare → „este un" — SavingsAccount moștenește BankAccount. SqlRepository implementează IRepository.
⚠️
Greșeala comună

Confundarea compoziției cu agregarea duce la bug-uri subtile. Dacă Order ar fi agregare față de OrderLine, liniile ar putea fi partajate între comenzi — ceea ce semantic nu are sens. Alege relația care reflectă corect domeniul.

05Clase și Metode Statice

Clasele și metodele statice sunt convenabile, dar creează cuplare ascunsă. Orice cod care apelează o metodă statică devine direct dependent de clasa respectivă — fără posibilitate de substituire sau testare.

❌ Cuplare ascunsă
DateTime.Now în interiorul logicii de business face codul netestabil — nu poți controla timpul în teste.
❌ Stare globală
Câmpuri statice mutable sunt stare globală — probleme cu concurența și cu izolarea testelor.
✅ Utilitare pure
Metode statice fără efecte secundare (Math.Abs, string.IsNullOrEmpty) sunt perfect acceptabile.
✅ Factory methods
Metode statice de creare (Order.CreateNew()) pot fi o interfață curată pentru construcție.
Înlocuirea dependenței statice cu abstracție C# 
// ❌ Netestabil — cum simulezi că e după ora 22:00?
public bool IsAfterHours() => DateTime.Now.Hour >= 22;

// ✅ Testabil — injectezi un IDateTimeProvider mock în teste
public interface IDateTimeProvider
{
    DateTime Now { get; }
}

public bool IsAfterHours(IDateTimeProvider clock)
    => clock.Now.Hour >= 22;

06Limitele OOP

OOP este un instrument excelent, dar nu singurul. Înainte de a continua cursul, este important să recunoaștem situațiile în care OOP poate deveni o problemă.

God Class O clasă care știe și face totul. OrderManager cu 800 de linii care validează, calculează, salvează în DB, trimite emailuri și generează rapoarte. Simptom: coeziune zero.
Anemic Domain Entitățile sunt simple structuri de date (doar proprietăți, fără comportament), iar toată logica stă în servicii externe. OOP formal, dar procedural în practică.
Over-engineering Ierarhii de 6 niveluri, interfețe pentru orice, abstractizări pentru cazuri ipotetice. YAGNI (You Aren't Gonna Need It) este un principiu, nu o slăbiciune.
Feature Envy O metodă care folosește mai mult datele altei clase decât ale clasei proprii. Semnalează că metoda ar trebui mutată în cealaltă clasă.
🧭
De reținut

OOP este mijlocul, nu scopul. Scopul este cod ușor de înțeles, de modificat și de testat. Dacă o soluție procedurală simplă rezolvă problema mai clar decât o ierarhie OOP complexă, soluția procedurală este mai bună.

07Exemplu Complet — Sistemul de Notificări

Un sistem care trimite notificări prin canale multiple (email, SMS, push). Vom aplica toți pilonii OOP și vom demonstra coeziunea și cuplarea mică.

NotificationSystem — toți pilonii OOP în acțiune

Abstractizare prin interfețe · Encapsulare a stării · Polimorfism la runtime · Compoziție în loc de moștenire

NotificationSystem.cs C# 
// ── Abstractizare: contracte clare, fără detalii de implementare ──
public interface INotificationChannel
{
    string Name { get; }
    bool CanSend(Notification notification);
    Task SendAsync(Notification notification);
}

public interface INotificationLogger
{
    Task LogAsync(string channel, Notification notification, bool success);
}

// ── Value Object: imutabil, egalitate prin valoare ──
public record Notification(
    string RecipientEmail,
    string RecipientPhone,
    string Subject,
    string Body,
    NotificationPriority Priority
);

public enum NotificationPriority { Low, Normal, High, Critical }

// ── Encapsulare: stare internă protejată, logică de business în clasă ──
public class EmailChannel : INotificationChannel
{
    private readonly IEmailProvider _provider; // dependență prin abstracție
    private int _sentCount = 0;              // stare internă, nu expusă direct

    public string Name => "Email";

    public EmailChannel(IEmailProvider provider)
        => _provider = provider ?? throw new ArgumentNullException(nameof(provider));

    public bool CanSend(Notification n)
        => !string.IsNullOrWhiteSpace(n.RecipientEmail);

    public async Task SendAsync(Notification n)
    {
        await _provider.SendEmailAsync(n.RecipientEmail, n.Subject, n.Body);
        _sentCount++; // starea se actualizează doar intern
    }

    public int GetSentCount() => _sentCount; // acces read-only controlat
}

public class SmsChannel : INotificationChannel
{
    private readonly ISmsProvider _provider;
    public string Name => "SMS";

    public SmsChannel(ISmsProvider provider) => _provider = provider;

    public bool CanSend(Notification n)
        => !string.IsNullOrWhiteSpace(n.RecipientPhone)
        && n.Priority >= NotificationPriority.High; // SMS doar pentru urgențe

    public async Task SendAsync(Notification n)
        => await _provider.SendSmsAsync(n.RecipientPhone, n.Body);
}

// ── Coeziune mare: NotificationDispatcher face un singur lucru ──
// ── Cuplare mică: depinde doar de interfețe, nu de tipuri concrete ──
public class NotificationDispatcher
{
    private readonly IReadOnlyList<INotificationChannel> _channels;
    private readonly INotificationLogger _logger;

    public NotificationDispatcher(
        IEnumerable<INotificationChannel> channels,
        INotificationLogger logger)
    {
        _channels = channels.ToList().AsReadOnly();
        _logger = logger;
    }

    public async Task DispatchAsync(Notification notification)
    {
        // Polimorfism: iterăm canale, fiecare știe singur dacă poate trimite
        var eligibleChannels = _channels.Where(c => c.CanSend(notification));

        foreach (var channel in eligibleChannels)
        {
            bool success = false;
            try
            {
                await channel.SendAsync(notification);
                success = true;
            }
            catch (Exception ex)
            {
                // izolarea erorilor: un canal căzut nu blochează celelalte
                await _logger.LogAsync(channel.Name, notification, false);
                continue;
            }
            await _logger.LogAsync(channel.Name, notification, success);
        }
    }
}

// ── Utilizare — cu DI container (ASP.NET Core) ──
// services.AddScoped<INotificationChannel, EmailChannel>();
// services.AddScoped<INotificationChannel, SmsChannel>();
// services.AddScoped<INotificationLogger, DatabaseNotificationLogger>();
// services.AddScoped<NotificationDispatcher>();
✓
Ce demonstrează acest exemplu

Abstractizare — interfețe pentru canale, provider-i, logger. Encapsulare — fiecare canal gestionează propriul contor și logica sa de eligibilitate. Polimorfism — CanSend și SendAsync funcționează diferit pentru fiecare canal. Compoziție — dispatcher-ul are canale, nu este un canal. Cuplare mică — toate dependențele sunt prin interfețe.
