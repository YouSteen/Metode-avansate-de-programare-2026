Design Patterns Creaționale

Pattern-urile creaționale se ocupă cu un singur lucru: cum se creează obiectele. Ele ascund decizia despre ce clasă concretă se instanțiază, permițând sistemului să fie configurat cu tipuri diferite fără a modifica codul care le folosește.

Obiective
  • Înțelegeți de ce crearea directă cu new poate deveni o problemă de design
  • Distingeți Factory Method de Abstract Factory și știți când să le alegeți
  • Aplicați Builder pentru construirea obiectelor cu mulți parametri opționali
  • Utilizați Prototype pentru clonarea obiectelor cu stare complexă
  • Cunoașteți implementările corecte ale Singleton și riscurile asociate

01Ce sunt design pattern-urile și de ce contează

Un design pattern este o soluție dovedită pentru o problemă recurentă de design. Nu este cod care se copiază, ci un vocabular comun și o structură de gândire. Când un coleg spune „folosim Factory Method aici", toată echipa înțelege imediat ce se întâmplă — fără să citească implementarea.

Catalogul GoF (Gang of Four — Gamma, Helm, Johnson, Vlissides, 1994) clasifică 23 de pattern-uri în trei categorii. Astăzi acoperim categoria creațională — cele cinci pattern-uri care controlează cum se creează obiectele.

Pattern Problema rezolvată Cuvânt cheie
Factory Method Subclasele decid ce tip concret se creează delegare
Abstract Factory Familie de obiecte compatibile, creare fără a specifica clasele concrete familie
Builder Construirea pas cu pas a obiectelor complexe construcție graduală
Prototype Clonarea unui obiect existent ca alternativă la construcție de la zero clonare
Singleton O singură instanță globală accesibilă din orice punct instanță unică
💡
De ce nu e suficient new?

Când scrieți new EmailNotifier() direct în codul de business, cuplați acel cod la o clasă concretă. Nu puteți testa fără a trimite e-mailuri reale, nu puteți schimba implementarea fără a modifica codul client, nu puteți varia comportamentul la runtime. Pattern-urile creaționale separă decizia de creare de codul care folosește obiectul.

02Factory Method

🏭
Factory Method
Definește o interfață pentru crearea unui obiect, dar lasă subclasele să decidă ce clasă concretă se instanțiază. Factory Method transferă responsabilitatea instanțierii către subclase.
Creațional GoF

Problema apare când o clasă de bază are nevoie să creeze obiecte, dar nu cunoaște tipul concret sau nu dorește să depindă de el. Clasa de bază definește o metodă abstractă — factory method — pe care fiecare subclasă o implementează cu tipul concret potrivit.

Exemplu — sistem de export rapoarte

Un ReportExporter de bază știe cum să pregătească și să livreze un raport, dar nu știe în ce format. Subclasele HtmlReportExporter și PdfReportExporter furnizează formatul lor prin factory method.

ReportExporter.cs — Factory Method C# 
// Clasa de bază: definește algoritmul, delega crearea obiectului
public abstract class ReportExporter
{
    // Factory Method — subclasele decid ce formatter se creează
    protected abstract IReportFormatter CreateFormatter();

    // Template Method: algoritmul de export rămâne aici
    public void Export(ReportData data)
    {
        var formatter = CreateFormatter(); // apel factory method
        var content   = formatter.Format(data);
        Deliver(content);
    }

    private void Deliver(string content)
        => File.WriteAllText($"report_{DateTime.Now:yyyyMMdd}.out", content);
}

public interface IReportFormatter
{
    string Format(ReportData data);
}

// Subclasa 1: creează un formatter HTML
public class HtmlReportExporter : ReportExporter
{
    protected override IReportFormatter CreateFormatter()
        => new HtmlFormatter();
}

// Subclasa 2: creează un formatter PDF
public class PdfReportExporter : ReportExporter
{
    protected override IReportFormatter CreateFormatter()
        => new PdfFormatter();
}

// Codul client nu știe ce formatter se folosește
ReportExporter exporter = GetExporterForCurrentConfig(); // returnează HTML sau PDF
exporter.Export(reportData); // același apel, comportament diferit
✓ Când se folosește
Când o clasă nu poate anticipa tipul de obiect pe care trebuie să îl creeze. Când subclasele trebuie să controleze crearea. Când doriți să centralizați logica de construire într-un singur loc.
✗ Când nu se potrivește
Când există un singur tip concret și nu va exista altul. Ierarhia de subclase poate deveni adâncă dacă nu se gestionează cu atenție — în aceste situații, compoziția cu o interfață directă e mai simplă.

03Abstract Factory

🏗
Abstract Factory
Furnizează o interfață pentru crearea unei familii de obiecte înrudite sau dependente, fără a specifica clasele lor concrete.
Creațional GoF

Dacă Factory Method creează un singur tip de obiect, Abstract Factory creează o familie de tipuri compatibile. Garanția esențială: toate obiectele create de aceeași factory sunt compatibile între ele.

Interacțiunea obiectelor — diagramă de secvență

Exemplu — sistem de notificări multi-canal

Un sistem de notificări care trebuie să funcționeze în două moduri: producție (notificări reale — e-mail, SMS) și testare (notificări simulate — scrise în consolă sau fișier). Abstract Factory garantează că întotdeauna se folosesc componente din același „mediu".

INotificationFactory.cs — Abstract Factory C# 
// Interfața factory-ului abstract — declară metodele de creare
public interface INotificationFactory
{
    IEmailSender  CreateEmailSender();
    ISmsSender    CreateSmsSender();
    IPushSender   CreatePushSender();
}

// Produsele abstracte
public interface IEmailSender { void Send(string to, string body); }
public interface ISmsSender   { void Send(string phone, string text); }
public interface IPushSender  { void Push(string deviceId, string msg); }

// ── Factory concretă 1: PRODUCȚIE ─────────────────────────────────────
public class ProductionNotificationFactory : INotificationFactory
{
    public IEmailSender CreateEmailSender() => new SmtpEmailSender();
    public ISmsSender   CreateSmsSender()   => new TwilioSmsSender();
    public IPushSender  CreatePushSender()  => new FirebasePushSender();
}

// ── Factory concretă 2: TESTARE ───────────────────────────────────────
public class FakeNotificationFactory : INotificationFactory
{
    public IEmailSender CreateEmailSender() => new InMemoryEmailSender();
    public ISmsSender   CreateSmsSender()   => new InMemorySmsSender();
    public IPushSender  CreatePushSender()  => new InMemoryPushSender();
}

// ── Codul client: nu știe cu ce factory lucrează ──────────────────────
public class AlertService
{
    private readonly IEmailSender _email;
    private readonly ISmsSender   _sms;

    public AlertService(INotificationFactory factory)
    {
        _email = factory.CreateEmailSender();
        _sms   = factory.CreateSmsSender();
    }

    public void SendAlert(string message)
    {
        _email.Send("[email protected]", message);
        _sms.Send("+40700000000", message);
    }
}

// Configurare: un singur loc unde se decide familia
INotificationFactory factory = Environment.IsProduction
    ? new ProductionNotificationFactory()
    : new FakeNotificationFactory();

var alerts = new AlertService(factory); // primește familia potrivită
✓
Garanția de compatibilitate

Abstract Factory garantează că niciodată nu se va folosi un SmtpEmailSender de producție împreună cu un InMemorySmsSender de testare. Familia este unitară prin construcție — nu prin convenție sau documentație.

✓ Când se folosește
Când sistemul trebuie să fie independent de modul în care sunt create produsele sale. Când doriți să garantați că obiectele dintr-o familie sunt compatibile. Când este necesară schimbarea întregii familii de obiecte dintr-un singur punct.
✗ Când nu se potrivește
Când familia are un singur produs — reprezintă o soluție supradimensionată față de un simplu Factory Method. Adăugarea unui produs nou în familia existentă necesită modificarea interfeței factory și a tuturor implementărilor concrete.

04Builder

🧱
Builder
Separă construirea unui obiect complex de reprezentarea sa. Același proces de construcție poate produce reprezentări diferite.
Creațional GoF

Problema apare când un obiect are mulți parametri, unii opționali, iar constructorul devine imposibil de utilizat corect — new Email(to, cc, bcc, subject, body, isHtml, priority, attachments, replyTo). Builder oferă o interfață fluentă care construiește obiectul pas cu pas, validând starea finală înainte de a o livra.

Interacțiunea obiectelor — diagramă de secvență

Exemplu — construirea unui e-mail

❌ Constructor cu prea mulți parametri — greu de citit și de utilizat corect
// Care parametru e cc și care e bcc? Care bool e isHtml?
var email = new Email(
    "[email protected]",
    "[email protected]",
    null,
    "Subiect",
    "Conținut",
    true,
    Priority.High,
    null
);
✅ Builder fluent — lizibil, sigur, extensibil
var email = new EmailBuilder()
    .To("[email protected]")
    .Cc("[email protected]")
    .Subject("Subiect")
    .Body("Conținut")
    .AsHtml()
    .WithPriority(Priority.High)
    .Build();
EmailBuilder.cs — implementare Builder fluent C# 
public class Email
{
    public string   To       { get; init; } = string.Empty;
    public string?  Cc       { get; init; }
    public string   Subject  { get; init; } = string.Empty;
    public string   Body     { get; init; } = string.Empty;
    public bool     IsHtml   { get; init; }
    public Priority Priority { get; init; } = Priority.Normal;
}

public class EmailBuilder
{
    private string?   _to, _cc, _subject, _body;
    private bool      _isHtml;
    private Priority  _priority = Priority.Normal;

    // Fiecare metodă returnează this — permite înlănțuirea
    public EmailBuilder To(string address)       { _to = address;  return this; }
    public EmailBuilder Cc(string address)       { _cc = address;  return this; }
    public EmailBuilder Subject(string subject)   { _subject = subject; return this; }
    public EmailBuilder Body(string body)         { _body = body;   return this; }
    public EmailBuilder AsHtml()                  { _isHtml = true; return this; }
    public EmailBuilder WithPriority(Priority p)  { _priority = p;  return this; }

    public Email Build()
    {
        // Validare înaintea construcției — eroare clară, nu NullReferenceException
        if (string.IsNullOrWhiteSpace(_to))
            throw new InvalidOperationException("Destinatarul (To) este obligatoriu");
        if (string.IsNullOrWhiteSpace(_subject))
            throw new InvalidOperationException("Subiectul este obligatoriu");

        return new Email
        {
            To       = _to!,
            Cc       = _cc,
            Subject  = _subject!,
            Body     = _body ?? string.Empty,
            IsHtml   = _isHtml,
            Priority = _priority
        };
    }
}
💡
Builder vs Record cu init — când să alegi

În C# modern, record cu with sau proprietăți init poate înlocui Builder-ul pentru obiecte simple. Builder rămâne relevant când există validare complexă înainte de construcție, când ordinea pașilor contează sau când construcția are ramuri condiționale (if isHtml → procesează template).

✓ Când se folosește
Când obiectul are mulți parametri, mulți opționali. Când construcția implică validare sau pași condiționali. Când doriți să preveniți construirea unui obiect în stare invalidă.
✗ Când nu se potrivește
Când obiectul este simplu și are 2–3 câmpuri obligatorii — un constructor obișnuit e mai clar. Risc de supraproiectare când obiectele sunt simple.

05Prototype

🧬
Prototype
Permite crearea de noi obiecte prin copierea unui obiect prototip existent, în loc de a le construi de la zero.
Creațional GoF

Prototype este util când crearea unui obiect de la zero este costisitoare (citire din bază de date, inițializare complexă) și aveți nevoie de mai multe instanțe similare. În loc să reconstruiți, clonați un obiect existent și modificați doar ce diferă.

Clonare superficială vs clonare profundă

Aceasta este distincția esențială a pattern-ului Prototype, sursă frecventă de erori atunci când este neglijată:

DocumentTemplate.cs — Prototype cu clonare profundă C# 
public class DocumentTemplate : ICloneable
{
    public string       Title    { get; set; } = string.Empty;
    public List<string> Sections { get; set; } = new();
    public PageLayout   Layout   { get; set; } = new();

    // ❌ Clonare superficială — Sections și Layout sunt referințe partajate
    public DocumentTemplate ShallowClone()
        => (DocumentTemplate)MemberwiseClone();

    // ✓ Clonare profundă — fiecare obiect clonat are propria copie a datelor
    public object Clone()
        => new DocumentTemplate
        {
            Title    = Title,                        // string e imutabil — OK
            Sections = new List<string>(Sections),  // copie nouă a listei
            Layout   = Layout.Clone()              // PageLayout se clonează la rândul ei
        };
}

// Demonstrarea problemei cu clonarea superficială
var original = new DocumentTemplate { Title = "Raport Q1" };
original.Sections.Add("Introducere");

var shallow = original.ShallowClone();
shallow.Sections.Add("Capitolul 2");
// ❌ original.Sections conține acum și "Capitolul 2" — aceeași referință!

var deep = (DocumentTemplate)original.Clone();
deep.Sections.Add("Capitolul 2");
// ✓ original.Sections nu e afectat — deep are propria sa listă

Registry de prototipuri

Un registru de prototipuri centralizează instanțele prototip și permite accesul la ele prin cheie. Clienții cer o copie după cheie — nu creează obiecte de la zero.

TemplateRegistry.cs — registru de prototipuri C# 
public class TemplateRegistry
{
    private readonly Dictionary<string, DocumentTemplate> _prototypes = new();

    public void Register(string key, DocumentTemplate template)
        => _prototypes[key] = template;

    public DocumentTemplate Get(string key)
    {
        if (!_prototypes.TryGetValue(key, out var proto))
            throw new KeyNotFoundException($"Prototip '{key}' neînregistrat");
        return (DocumentTemplate)proto.Clone(); // returnează o copie, nu prototipul
    }
}

// Utilizare: înregistrăm prototipurile o dată, la inițializare
var registry = new TemplateRegistry();
registry.Register("raport-trimestrial", CreateQuarterlyTemplate());
registry.Register("factura",            CreateInvoiceTemplate());

// Ulterior, creăm copii personalizate fără să reconstruim de la zero
var doc = registry.Get("raport-trimestrial");
doc.Title = "Raport Q2 2026"; // modificăm copia, prototipul rămâne intact
✓ Când se folosește
Când instanțierea este costisitoare și aveți nevoie de mai multe copii similare. Când clasa exactă a obiectului nu se cunoaște la compile-time. Când doriți să variați configurația unui obiect de bază.
✗ Când nu se potrivește
Când obiectele au referințe circulare — clonarea profundă devine complicată. Când obiectele sunt imutabile — nu are sens să le clonați dacă nu le modificați.

06Singleton

🔒
Singleton
Asigură că o clasă are o singură instanță și furnizează un punct global de acces la aceasta.
Creațional GoF

Singleton este cel mai cunoscut pattern creațional și, totodată, cel mai frecvent utilizat greșit. Cazurile legitime sunt rare: configurație globală imutabilă, cache partajat, conexiune la o resursă care nu poate fi duplicată. În marea majoritate a situațiilor, o instanță înregistrată ca Singleton într-un container IoC este o soluție mai bună decât un Singleton static.

Implementări în C#

Singleton.cs — trei variante, de la problematică la recomandată C# 
// ── Varianta 1: Clasică, nesigură la multithreading ───────────────────
public class ConfigurationManager_V1
{
    private static ConfigurationManager_V1? _instance;

    private ConfigurationManager_V1() { }

    public static ConfigurationManager_V1 Instance
    {
        get
        {
            // ❌ Dacă două fire de execuție ajung simultan aici,
            // se pot crea două instanțe
            if (_instance == null)
                _instance = new ConfigurationManager_V1();
            return _instance;
        }
    }
}

// ── Varianta 2: Thread-safe cu lock — funcționează, dar lock e costisitor ─
public class ConfigurationManager_V2
{
    private static ConfigurationManager_V2? _instance;
    private static readonly object _lock = new();

    private ConfigurationManager_V2() { }

    public static ConfigurationManager_V2 Instance
    {
        get
        {
            lock (_lock) { // lock la fiecare acces — overhead inutil după inițializare
                if (_instance == null)
                    _instance = new ConfigurationManager_V2();
                return _instance;
            }
        }
    }
}

// ── Varianta 3: Lazy<T> — recomandată în C# ───────────────────────────
// Thread-safe prin garantarea CLR, fără lock explicit, lazy by default
public class ConfigurationManager
{
    private static readonly Lazy<ConfigurationManager> _lazy
        = new(() => new ConfigurationManager());

    private ConfigurationManager()
    {
        // Inițializare o singură dată — citire configurație, etc.
    }

    public static ConfigurationManager Instance => _lazy.Value;

    public string Get(string key) => /* citire din config */ string.Empty;
}
⚠️
Singleton și testabilitatea

Un Singleton static face testarea dificilă: nu îl puteți înlocui cu un mock, starea lui persistă între teste (producând dependențe între teste), iar testele nu pot rula în paralel în siguranță. Soluția preferabilă: înregistrați clasa ca AddSingleton în containerul IoC — obțineți o singură instanță per aplicație, dar injectabilă și înlocuibilă în teste.

✓ Când se justifică
Configurație globală read-only încărcată o singură dată. Cache partajat cu gestionare explicită a concurenței. Resursă care nu poate exista în mai multe instanțe prin natura ei (ex: conexiune hardware unică).
✗ Semnale de abuz
Folosit pentru a evita injectarea dependențelor. Stare mutabilă globală — produce bug-uri de concurență greu de depistat. Orice situație în care un container IoC cu AddSingleton ar rezolva problema mai elegant.

07Comparație și ghid de alegere

Cele cinci pattern-uri rezolvă probleme diferite. Ghidul de mai jos ajută la alegerea rapidă a celui potrivit:

Întrebare Pattern recomandat
Vreau să las subclasele să decidă ce tip concret se creează Factory Method
Am mai multe obiecte care trebuie să fie compatibile între ele și vreau să schimb familia dintr-un singur loc Abstract Factory
Obiectul meu are mulți parametri opționali și construcția poate eșua (validare) Builder
Construcția de la zero e costisitoare și am nevoie de copii ușor personalizate Prototype
Există o resursă care trebuie să aibă o singură instanță în toată aplicația Singleton sau AddSingleton în IoC
🧭
Pattern-urile nu se exclud

Este frecvent să combinați pattern-uri: un Abstract Factory care creează obiecte construite intern cu un Builder, sau un Prototype Registry care returnează copii ale unor obiecte create inițial prin Factory Method. Pattern-urile sunt unelte componibile, nu categorii exclusive.

08Laborator 5

Proiect independent de la zero, care aplică explicit pattern-urile creaționale. Tema: un sistem de generare documente — rapoarte și facturi — cu suport pentru mai multe formate de export și configurare flexibilă a conținutului.

🔬

Sistem de Generare Documente

Laborator 5 — aplicație .NET care aplică Factory Method, Abstract Factory, Builder și Prototype

Sistemul generează documente de două tipuri (rapoarte și facturi) în două formate (HTML și Plain Text). Fiecare document are un antet, secțiuni de conținut și un subsol. Documentele se construiesc din șabloane pre-configurate care se clonează și personalizează.

Cerințe
Factory Method Cerința 1 — Exportatori de documente
  • Definiți clasa abstractă DocumentExporter cu factory method CreateRenderer() și metoda Export(DocumentData data) care apelează renderer-ul
  • Implementați HtmlDocumentExporter și PlainTextDocumentExporter — fiecare creează renderer-ul specific în CreateRenderer()
  • IDocumentRenderer expune metoda Render(DocumentData data): string
  • Codul client primește un DocumentExporter prin injecție și apelează Export() fără să știe ce renderer se folosește
Abstract Factory Cerința 2 — Familie de componente pentru documente
  • Definiți interfața IDocumentComponentFactory cu metodele: CreateHeader(), CreateSection(), CreateFooter()
  • Implementați ReportComponentFactory (antet cu logo și număr de raport, secțiuni cu tabel de date, subsol cu semnătură) și InvoiceComponentFactory (antet cu date fiscale, secțiuni cu linii de factură, subsol cu total)
  • Clasa DocumentAssembler primește IDocumentComponentFactory prin constructor și asamblează documentul folosind componentele furnizate — nu știe dacă e raport sau factură
Builder Cerința 3 — Configurarea metadatelor documentului
  • Clasa DocumentData conține: titlu, autor, dată, lista de secțiuni, format pagină, orientare, note de subsol (opționale)
  • Implementați DocumentDataBuilder cu interfață fluentă: .WithTitle(), .ByAuthor(), .WithSection(), .InLandscape(), .WithFootnote(), .Build()
  • Metoda Build() validează că titlul și autorul sunt completate și că există cel puțin o secțiune — aruncă InvalidOperationException altfel
Prototype Cerința 4 — Registru de șabloane
  • Clasa DocumentTemplate conține configurația unui tip de document: titlu implicit, secțiuni predefinite, setări de formatare
  • Implementați clonarea profundă pe DocumentTemplate — lista de secțiuni și obiectul de formatare trebuie să fie copii independente
  • Implementați TemplateRegistry care înregistrează prototipuri după cheie și returnează clonări la cerere
  • Demonstrați că modificarea unui document clonat nu afectează șablonul original din registru — scrieți un test NUnit pentru aceasta
Singleton Cerința 5 — Configurație globală
  • Implementați AppConfiguration ca Singleton cu Lazy<T> — conține setări globale: director de ieșire, format implicit, autor implicit
  • Configurația se încarcă o singură dată, la primul acces, dintr-un fișier appsettings.json sau din valori implicite
  • Justificați în README.md de ce Singleton este adecvat pentru configurație și când ar trebui înlocuit cu AddSingleton în IoC
Teste NUnit Cerința 6 — Testare
  • Test care verifică că HtmlDocumentExporter și PlainTextDocumentExporter produc output diferit pentru aceleași date (Factory Method)
  • Test care verifică că DocumentAssembler produce un document cu header de raport când primește ReportComponentFactory și header de factură când primește InvoiceComponentFactory (Abstract Factory)
  • Test care verifică că DocumentDataBuilder.Build() aruncă excepție când titlul lipsește (Builder — validare)
  • Test care verifică că două clonări din același prototip returnează obiecte independente — modificarea unuia nu afectează celălalt (Prototype)

💬 Teme de Discuție
  1. Factory Method se bazează pe moștenire, Abstract Factory pe compoziție. Care din cele două abordări este mai flexibilă pe termen lung și de ce?

  3. Prototype presupune că știți cum să clonați corect obiectul. Ce se întâmplă cu referințele la obiecte imutabile (string, record)? Trebuie clonate?
  4. Singleton este frecvent descris ca un anti-pattern. Care sunt argumentele pro și contra și în ce condiții este justificat față de AddSingleton în IoC?
