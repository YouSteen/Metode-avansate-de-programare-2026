Principiile SOLID — Partea I

Trei dintre cele cinci principii care transformă codul corect în cod bine proiectat: Single Responsibility, Open/Closed și Liskov Substitution. Nu sunt reguli arbitrare — fiecare rezolvă o problemă reală de mentenanță.

🎯 Obiective
  • Înțelegerea originii și scopului principiilor SOLID — probleme reale, nu teorie abstractă
  • Aplicarea SRP: identificarea și separarea responsabilităților dintr-o clasă
  • Aplicarea OCP: extinderea comportamentului fără a modifica codul existent
  • Aplicarea LSP: garantarea că subclasele sunt substituibile fără surprize

01Introducere în SOLID

Robert C. Martin (Uncle Bob) a formalizat aceste principii la începutul anilor 2000, pornind de la observații acumulate în decenii de practică. Acronimul SOLID a fost propus ulterior de Michael Feathers.

Scopul lor comun este simplu: cod care rezistă în timp. Cod pe care îl poți extinde fără să spargi ce era deja funcțional, cod pe care îl poți testa în izolare, cod pe care un coleg îl poate înțelege fără să îți ceară explicații.

Principiile SOLID nu sunt reguli de urmat orbește. Sunt instrumente de gândire care te ajută să recunoști când un design va deveni o problemă mai târziu.

În acest curs acoperim primele trei. Celelalte două — ISP și DIP — sunt subiectul Cursului 4.

Capcana comună

SOLID poate deveni o sursă de over-engineering dacă este aplicat mecanic. Regula: aplică un principiu când există deja o problemă sau când schimbarea viitoare este previzibilă și concretă, nu ipotetică.

02SRP — Single Responsibility Principle

S
Single Responsibility Principle
„O clasă trebuie să aibă un singur motiv de schimbare."

Definiția originală vorbește despre motive de schimbare, nu despre numărul de metode sau linii de cod. Un motiv de schimbare vine întotdeauna de la un actor — o persoană sau un sistem care poate cere modificarea clasei.

O clasă care servește doi actori diferiți are două motive de schimbare și va trebui modificată din două direcții independente — ceea ce crește riscul că o modificare pentru un actor o va strica pe cealaltă.

Semne că SRP este încălcat

⚠ Nume vagi: OrderManager, UserHelper, DataUtil — când nu poți numi precis ce face o clasă, face prea multe lucruri.
⚠ Metode fără legătură: aceeași clasă are metode pentru validare, calcul, persistență și trimitere email — patru responsabilități, patru motive de schimbare.
⚠ Clase mari: o clasă cu 400+ linii este aproape sigur un semnal că face mai mult decât ar trebui.
⚠ Importuri mixte: o clasă care importă simultan System.Data, System.Net.Mail și System.IO amestecă infrastructuri care nu au ce căuta împreună.

Exemplu: God Class → clase cu responsabilități clare

Pornim de la un OrderProcessor care face totul: validează comanda, calculează prețul, salvează în baza de date și trimite email de confirmare. Aceasta este o clasă cu patru responsabilități și patru actori potențiali: echipa de business (reguli de validare), contabilitate (calcul prețuri), DBA (schema bazei de date) și echipa de comunicare (template email).

❌ Înainte — o clasă, patru responsabilități
public class OrderProcessor
{
    // Responsabilitate 1: Validare
    public bool ValidateOrder(Order order)
    {
        if (order.Items.Count == 0) return false;
        if (order.Customer == null) return false;
        return true;
    }

    // Responsabilitate 2: Calcul preț
    public decimal CalculateTotal(Order order)
    {
        var total = order.Items.Sum(i => i.Price * i.Quantity);
        if (order.Customer.IsPremium) total *= 0.9m;
        return total;
    }

    // Responsabilitate 3: Persistență
    public void SaveOrder(Order order)
    {
        using var conn = new SqlConnection("Server=...");
        // INSERT INTO Orders...
    }

    // Responsabilitate 4: Notificare
    public void SendConfirmation(Order order)
    {
        var smtp = new SmtpClient("smtp.company.com");
        // send email...
    }
}
✅ După — fiecare clasă are o singură responsabilitate
// Responsabilitate 1: Validare comenzi
public class OrderValidator
{
    public bool IsValid(Order order)
    {
        if (order.Items.Count == 0) return false;
        if (order.Customer == null) return false;
        return true;
    }
}

// Responsabilitate 2: Calcul prețuri
public class OrderPriceCalculator
{
    public decimal Calculate(Order order)
    {
        var total = order.Items.Sum(i => i.Price * i.Quantity);
        if (order.Customer.IsPremium) total *= 0.9m;
        return total;
    }
}

// Responsabilitate 3: Persistență
public class OrderRepository
{
    public void Save(Order order) { /* SQL */ }
}

// Responsabilitate 4: Notificări
public class OrderNotificationService
{
    public void SendConfirmation(Order order) { /* SMTP */ }
}

// Orchestrator — coordonează fără să implementeze
public class OrderProcessor
{
    private readonly OrderValidator          _validator;
    private readonly OrderPriceCalculator     _calculator;
    private readonly OrderRepository          _repository;
    private readonly OrderNotificationService _notifier;

    public OrderProcessor(
        OrderValidator          validator,
        OrderPriceCalculator     calculator,
        OrderRepository          repository,
        OrderNotificationService notifier)
    {
        _validator  = validator;
        _calculator = calculator;
        _repository = repository;
        _notifier   = notifier;
    }

    public void Process(Order order)
    {
        if (!_validator.IsValid(order))
            throw new InvalidOrderException();

        order.Total = _calculator.Calculate(order);
        _repository.Save(order);
        _notifier.SendConfirmation(order);
    }
}
✓
Ce câștigăm

Dacă echipa de business schimbă regulile de validare, modificăm doar OrderValidator. OrderRepository și OrderNotificationService nu sunt atinse. Fiecare clasă poate fi testată independent, în izolare completă.

💡
SRP se aplică și la nivel de metodă

O metodă care face mai mult de un lucru este un candidat la extracție. Semnalul: dacă simți nevoia unui comentariu care separă blocuri de cod dintr-o metodă, acele blocuri sunt metode separate.

03OCP — Open/Closed Principle

O
Open/Closed Principle
„Entitățile software trebuie să fie deschise pentru extensie, dar închise pentru modificare."

Formulat de Bertrand Meyer în 1988, OCP spune că atunci când trebuie să adaugi un comportament nou, nu ar trebui să modifici codul care funcționează deja — ci să îl extinzi prin noi clase sau implementări.

Motivul este practic: codul existent este testat și stabil. Orice modificare în el introduce riscul unui regres. OCP ne îndeamnă să proiectăm sisteme în care extinderea este posibilă fără a atinge codul testat.

⚠️
OCP și YAGNI — când se aplică

YAGNI (You Aren't Gonna Need It) spune: nu construi abstractizări înainte să ai nevoie de ele. OCP și YAGNI nu se contrazic — se completează. Regula practică: aplică OCP când schimbarea este concretă și previzibilă (știi că vor mai veni tipuri noi), nu când este ipotetică. Dacă ai un singur format de export și nu știi dacă vor fi altele, nu crea o interfață „pentru viitor" — adaug-o când apare al doilea caz.

Exemplul 1 — if/switch pe tip: DiscountCalculator

Cel mai frecvent semnal că OCP este încălcat: un bloc if sau switch care verifică tipul unui obiect pentru a decide comportamentul. La fiecare nou tip, trebuie să modifici clasa existentă.

❌ Orice discount nou înseamnă modificarea clasei existente
public class DiscountCalculator
{
    public decimal Calculate(Order order, string customerType)
    {
        if (customerType == "Regular")
            return order.Total * 0.05m;

        if (customerType == "Premium")
            return order.Total * 0.10m;

        if (customerType == "VIP")
            return order.Total * 0.20m;

        // Cerință nouă: SeasonalDiscount?
        // → trebuie să modific această metodă
        return 0;
    }
}
✅ Interfață + injecție — tip nou = clasă nouă, nimic nu se modifică
public interface IDiscountStrategy
{
    decimal Calculate(Order order);
}

public class RegularDiscount : IDiscountStrategy
{
    public decimal Calculate(Order o) => o.Total * 0.05m;
}

public class PremiumDiscount : IDiscountStrategy
{
    public decimal Calculate(Order o) => o.Total * 0.10m;
}

public class VipDiscount : IDiscountStrategy
{
    public decimal Calculate(Order o) => o.Total * 0.20m;
}

// Cerință nouă: adaugi SeasonalDiscount : IDiscountStrategy
// DiscountCalculator și testele existente rămân NEATINSE
public class DiscountCalculator
{
    private readonly IDiscountStrategy _strategy;

    // strategia este injectată — nu hardcodată
    public DiscountCalculator(IDiscountStrategy strategy)
        => _strategy = strategy;

    public decimal Calculate(Order order)
        => _strategy.Calculate(order);
}

// Utilizare:
var calc = new DiscountCalculator(new VipDiscount());
var discount = calc.Calculate(order);
✓
Ce câștigăm

Adăugarea unui nou tip de discount înseamnă o clasă nouă. DiscountCalculator și testele sale existente nu sunt atinse. IDiscountStrategy este injectată prin constructor — calculatorul nu știe și nu îi pasă ce strategie primește. Acesta este Strategy pattern — îl vom studia detaliat la Cursul 8.

Exemplul 2 — clasă abstractă + injecție: exportul de rapoarte

Un alt scenariu frecvent: un serviciu care generează rapoarte și le exportă. Fiecare format nou (HTML, PDF, CSV) forțează modificarea clasei centrale. Soluția folosește o clasă abstractă pentru comportamentul comun și metode abstracte pentru pașii variabili — combinat cu injecție de dependențe.

❌ ReportExporter știe despre fiecare format — OCP încălcat
public class ReportExporter
{
    public void Export(Report report, string format)
    {
        // logică comună: validare, logging...
        Console.WriteLine($"Exporting: {report.Title}");

        switch (format)
        {
            case "html":
                // generează HTML...
                break;
            case "pdf":
                // generează PDF...
                break;
            case "csv":
                // generează CSV...
                break;
            // format nou → modific această clasă
        }
    }
}
✅ Clasă abstractă + injecție — format nou = subclasă nouă
// Comportamentul comun este în clasa abstractă (ÎNCHIS)
public abstract class ReportExporter
{
    // Template Method: scheletul algoritmului — nu se modifică
    public void Export(Report report)
    {
        Validate(report);
        Console.WriteLine($"Exporting: {report.Title}");
        WriteContent(report);  // pasul variabil
    }

    private void Validate(Report r)
    {
        if (r == null) throw new ArgumentNullException(nameof(r));
    }

    // Pasul variabil — subclasele îl implementează (DESCHIS)
    protected abstract void WriteContent(Report report);
}

public class HtmlReportExporter : ReportExporter
{
    protected override void WriteContent(Report r)
        => Console.WriteLine($"<html>{r.Content}</html>");
}

public class CsvReportExporter : ReportExporter
{
    protected override void WriteContent(Report r)
        => Console.WriteLine($"title,content\n{r.Title},{r.Content}");
}

// Format nou: adaugi PdfReportExporter : ReportExporter
// ReportExporter (baza) rămâne NEATINS

// Serviciu care primește exporterul prin injecție:
public class ReportService
{
    private readonly ReportExporter _exporter;

    public ReportService(ReportExporter exporter)
        => _exporter = exporter;

    public void GenerateAndExport(Report report)
        => _exporter.Export(report);
}

// Utilizare — schimbi formatul fără să atingi ReportService:
var service = new ReportService(new HtmlReportExporter());
service.GenerateAndExport(report);

var csvService = new ReportService(new CsvReportExporter());
csvService.GenerateAndExport(report);
💡
Template Method Pattern

Clasa abstractă ReportExporter definește scheletul algoritmului în Export() — validare, logging, scriere conținut — iar pasul variabil (WriteContent) este delegat subclaselor. Comportamentul comun este închis în clasa de bază; comportamentul specific formatului este deschis extensiei prin suprascrierea metodei abstracte. Acesta este Template Method pattern — îl vom studia la Cursul 8.

Alternativa: compoziție în loc de moștenire

Template Method rezolvă problema, dar o face prin moștenire — subclasele sunt cuplate structural la clasa de bază. GoF recomandă să favorizezi compoziția față de moștenire. Varianta prin compoziție: ReportExporter devine o clasă concretă care primește un IContentWriter injectat. Niciun abstract, nicio ierarhie.

Template Method — extensie prin moștenire
// Subclasa este LEGATĂ de clasa de bază
// Nu poți schimba formatul la runtime
// Nu poți combina doi scriitori
public abstract class ReportExporter
{
    public void Export(Report report)
    {
        Validate(report);
        WriteContent(report); // apel virtual
    }
    protected abstract void WriteContent(Report r);
}

public class HtmlReportExporter : ReportExporter
{
    protected override void WriteContent(Report r)
        => Console.WriteLine($"<html>{r.Content}</html>");
}

// Utilizare — formatul e fix la compile-time:
var exporter = new HtmlReportExporter();
exporter.Export(report);
✅ Compoziție — IContentWriter injectat, fără ierarhie
// Contract pentru pasul variabil
public interface IContentWriter
{
    void Write(Report report);
}

public class HtmlContentWriter : IContentWriter
{
    public void Write(Report r)
        => Console.WriteLine($"<html>{r.Content}</html>");
}

public class CsvContentWriter : IContentWriter
{
    public void Write(Report r)
        => Console.WriteLine($"title,content\n{r.Title},{r.Content}");
}

// Clasa concretă — primește scriitorul prin constructor
public class ReportExporter
{
    private readonly IContentWriter _writer;

    public ReportExporter(IContentWriter writer)
        => _writer = writer;

    public void Export(Report report)
    {
        if (report == null) throw new ArgumentNullException(nameof(report));
        Console.WriteLine($"Exporting: {report.Title}");
        _writer.Write(report); // delegare, nu moștenire
    }
}

// Utilizare — formatul se alege la runtime:
var exporter = new ReportExporter(new HtmlContentWriter());
exporter.Export(report);

// Schimbi formatul fără subclasare:
var csvExporter = new ReportExporter(new CsvContentWriter());
csvExporter.Export(report);
🔀
Moștenire vs. Compoziție — diferențele practice

Template Method (moștenire): formatul este fix la compile-time; subclasa este cuplată la clasa de bază; nu poți combina doi scriitori într-o singură instanță.

Compoziție: scriitorul poate fi schimbat la runtime; ReportExporter nu știe nimic despre formatele concrete; poți injecta un CompositeWriter care scrie în mai multe formate simultan — fără nicio modificare în ReportExporter.

În practică: preferă compoziția când comportamentul variabil trebuie ales la runtime sau combinat. Folosește moștenirea când există un algoritm cu structură fixă și variații minore la pași discreți.

Interfață vs. clasă abstractă — când alegi ce

Situație Folosești
Implementări complet diferite, niciun cod comun interface
Există comportament comun + pași variabili abstract class
O clasă trebuie să respecte mai multe contracte interface (C# permite multiple)
Comportament implicit opțional pentru unele metode interface + default implementation (C# 8+)

04LSP — Liskov Substitution Principle

L
Liskov Substitution Principle
„Dacă S este subtip al lui T, obiectele de tip T pot fi înlocuite cu obiecte de tip S fără a altera corectitudinea programului."

Formulat de Barbara Liskov în 1987, LSP spune că o subclasă trebuie să se comporte conform contractului clasei de bază. Codul care funcționează cu tipul de bază trebuie să funcționeze corect și cu orice subtip, fără a ști că este subtip.

Tradus practic: dacă trebuie să verifici cu if (obj is SubclassType) pentru a trata un subtip diferit, LSP este probabil încălcat.

Exemplul clasic: Rectangle și Square

Matematic, un pătrat este un dreptunghi. Dar din punct de vedere OOP, nu neapărat — depinde de comportament.

❌ Square extinde Rectangle — LSP încălcat
public class Rectangle
{
    public virtual int Width  { get; set; }
    public virtual int Height { get; set; }
    public int Area() => Width * Height;
}

public class Square : Rectangle
{
    // Forțăm invarianta: latura = latura
    public override int Width
    {
        set { base.Width = value; base.Height = value; }
    }
    public override int Height
    {
        set { base.Width = value; base.Height = value; }
    }
}

// Cod care funcționează cu Rectangle, se sparge cu Square:
void SetDimensions(Rectangle r)
{
    r.Width  = 4;
    r.Height = 5;
    // Așteptăm: Area() == 20
    // Cu Square: Area() == 25 ← surpriză!
}
✅ Interfețe separate — fără ierarhie forțată
public interface IShape
{
    int Area();
}

public class Rectangle : IShape
{
    public int Width  { get; }
    public int Height { get; }

    public Rectangle(int w, int h)
        => (Width, Height) = (w, h);

    public int Area() => Width * Height;
}

public class Square : IShape
{
    public int Side { get; }

    public Square(int side) => Side = side;

    public int Area() => Side * Side;
}

// Ambele sunt IShape → substituibile corect
IShape s1 = new Rectangle(4, 5); // Area = 20
IShape s2 = new Square(5);      // Area = 25

Exemplu practic C#: păsări și zbor

Un exemplu mai apropiat de cod real: o ierarhie de animale unde subclasa este forțată să implementeze un comportament pe care nu îl poate asigura.

❌ Penguin aruncă excepție pentru Fly() — contract rupt
public class Bird
{

public class Eagle : Bird

public interface IBird
{

}

public interface IFlyable
{

}

Design prin Contract și LSP

Design by Contract (DbC) este o tehnică introdusă de Bertrand Meyer care formalizează exact ce înseamnă „contractul" unui tip. Fiecare metodă are trei componente:

P
Precondiție — ce trebuie să fie adevărat înainte de apel
Responsabilitatea apelantului. Dacă precondiția nu e respectată, metoda nu garantează nimic. LSP spune: subclasa nu poate întări precondiția — nu poate cere mai mult decât baza.
Q
Postcondiție — ce garantează metoda după execuție
Responsabilitatea implementatorului. LSP spune: subclasa nu poate slăbi postcondiția — trebuie să garanteze cel puțin ce garantează baza.
I
Invariantă — ce rămâne adevărat pe tot ciclul de viață al obiectului
LSP spune: subclasa trebuie să păstreze invariantele bazei. Dacă baza garantează că un câmp este întotdeauna pozitiv, subclasa nu îl poate lăsa negativ.

DbC în C# — Guard clauses și excepții explicite

C# nu are suport nativ pentru DbC ca Eiffel, dar putem aplica aceleași idei prin guard clauses la intrarea în metode și postcondiții verificate explicit. Important: contractul trebuie documentat la nivelul interfeței/clasei de bază — subclasele îl moștenesc și trebuie să îl respecte.

IDiscountCalculator.cs — contractul definit pe interfață C# 
/// Calculează discountul pentru o comandă.
/// Precondiție:  order != null, order.Total >= 0
/// Postcondiție: rezultatul este în intervalul [0, order.Total]
/// Invariantă:   rata de discount nu se poate schimba după construcție
public interface IDiscountCalculator
{
    decimal Calculate(Order order);
}

// Clasă de bază abstractă care aplică contractul —
// subclasele implementează doar logica, nu re-validează
public abstract class DiscountCalculatorBase : IDiscountCalculator
{
    public decimal Calculate(Order order)
    {
        // Precondiție — guard clause, verificată o singură dată în bază
        if (order == null)
            throw new ArgumentNullException(nameof(order));
        if (order.Total < 0)
            throw new ArgumentException("Total cannot be negative");

        var result = ComputeDiscount(order); // delegat subclaselor

        // Postcondiție — verificată tot în bază, nu în subclase
        if (result < 0 || result > order.Total)
            throw new InvalidOperationException(
                $"Discount {result} outside valid range [0, {order.Total}]");

        return result;
    }

    // Subclasele implementează DOAR logica — contractul e garantat de bază
    protected abstract decimal ComputeDiscount(Order order);
}

// Implementare corectă — respectă contractul implicit
public class PremiumDiscount : DiscountCalculatorBase
{
    protected override decimal ComputeDiscount(Order order)
        => order.Total * 0.10m; // întotdeauna în [0, Total] ✓
}

// Implementare greșită — ar fi prinsă de postcondiție la runtime
public class BuggyDiscount : DiscountCalculatorBase
{
    protected override decimal ComputeDiscount(Order order)
        => order.Total * 2.0m; // InvalidOperationException → LSP încălcat detectat!
}
✓
De ce acest pattern ajută LSP

Contractul este definit și verificat o singură dată, în clasa de bază. Subclasele nu pot ignora precondițiile (nu le mai văd) și nu pot produce rezultate care violează postcondițiile (baza le detectează). O subclasă defectă este prinsă la runtime cu un mesaj explicit, nu tăcut ca un bug logic.

DbC cu invariante — exemplu cu cont bancar

Invariantele sunt condițiile care trebuie să rămână adevărate pe tot ciclul de viață al obiectului. Un cont bancar are invarianta că soldul nu poate fi negativ (în varianta simplă). Orice subclasă — SavingsAccount, CreditAccount — trebuie să păstreze această invariantă sau să o extindă corect.

BankAccount.cs — invariantă garantată de baza C# 
public abstract class BankAccount
{
    protected decimal _balance;

    public decimal Balance => _balance;

    // Invariantă: soldul nu poate fi negativ după orice operație
    protected void CheckInvariant()
    {
        if (_balance < 0)
            throw new InvalidOperationException(
                "Invariant violated: balance cannot be negative");
    }

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("Amount must be positive");
        _balance += amount;
        CheckInvariant(); // invarianta verificată după fiecare operație
    }

    public virtual void Withdraw(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("Amount must be positive");
        if (amount > _balance) throw new InvalidOperationException("Insufficient funds");
        _balance -= amount;
        CheckInvariant();
    }
}

// SavingsAccount adaugă o restricție suplimentară (sold minim)
// dar nu slăbește invarianta bazei — LSP respectat
public class SavingsAccount : BankAccount
{
    private const decimal MinBalance = 50m;

    public override void Withdraw(decimal amount)
    {
        // Precondiție mai restrictivă? Nu — e o regulă de business
        // SavingsAccount garantează în plus: balance >= 50 după retragere
        if (_balance - amount < MinBalance)
            throw new InvalidOperationException(
                $"Must maintain minimum balance of {MinBalance}");
        base.Withdraw(amount); // refolosește logica bazei + CheckInvariant()
    }
}

// CreditAccount permite sold negativ — invarianta bazei este SCHIMBATĂ.
// BankAccount garantează: balance >= 0 după orice operație.
// CreditAccount are nevoie de: balance >= -creditLimit.
// Cele două invariante sunt incompatibile — CreditAccount NU poate
// extinde BankAccount fără să violeze LSP. Vezi soluția de mai jos.
⚠️
De ce CreditAccount nu poate extinde BankAccount

Orice cod care lucrează cu un BankAccount se poate baza pe invarianta balance >= 0 — de exemplu un serviciu de raportare care calculează expunerea totală. Dacă înlocuim BankAccount cu un CreditAccount al cărui sold poate fi -500, raportul produce valori incorecte fără nicio eroare. Acesta este exact scenariul pe care LSP îl interzice: substituția silențioasă care schimbă comportamentul.

Soluția corectă este să extragem contractul comun într-o interfață și să lăsăm cele două tipuri de cont să fie peer-uri, nu relație părinte-copil. Fiecare implementează interfața cu propriile invariante, fără să le moștenească pe ale celuilalt.

Refactorizare — interfețe granulare în loc de moștenire C# 
// Contractul comun: ce știe să facă orice cont
public interface IAccount
{
    Guid    Id      { get; }
    decimal Balance { get; }
    void    Deposit(decimal amount);
}

// Contractul pentru conturi care pot retrage (fără a specifica limita)
public interface IWithdrawable : IAccount
{
    /// Precondiție:  amount > 0
    /// Postcondiție: Balance scade cu amount (în limita permisă de implementare)
    void Withdraw(decimal amount);
}

// ── Implementare 1: DebitAccount ──────────────────────────────────────
// Invariantă proprie: Balance >= 0 întotdeauna
public class DebitAccount : IWithdrawable
{
    private decimal _balance;
    public Guid    Id      { get; } = Guid.NewGuid();
    public decimal Balance => _balance;

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("Amount must be positive");
        _balance += amount;
    }

    public void Withdraw(decimal amount)
    {
        if (amount <= 0)         throw new ArgumentException("Amount must be positive");
        if (amount > _balance)  throw new InvalidOperationException("Insufficient funds");
        _balance -= amount;
        // Invariantă: _balance >= 0 — garantată de verificarea de mai sus
    }
}

// ── Implementare 2: CreditAccount ────────────────────────────────────
// Invariantă proprie: Balance >= -CreditLimit  (poate fi negativ!)
public class CreditAccount : IWithdrawable
{
    private decimal _balance;
    private readonly decimal _creditLimit;

    public Guid    Id          { get; } = Guid.NewGuid();
    public decimal Balance     => _balance;
    public decimal CreditLimit => _creditLimit;
    public decimal Available   => _balance + _creditLimit;

    public CreditAccount(decimal creditLimit)
    {
        if (creditLimit <= 0) throw new ArgumentException("Credit limit must be positive");
        _creditLimit = creditLimit;
    }

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("Amount must be positive");
        _balance += amount;
    }

    public void Withdraw(decimal amount)
    {
        if (amount <= 0)            throw new ArgumentException("Amount must be positive");
        if (amount > Available)    throw new InvalidOperationException("Credit limit exceeded");
        _balance -= amount;
        // Invariantă proprie: _balance >= -_creditLimit — garantată mai sus
        // Nu moștenește invarianta DebitAccount (balance >= 0) — și nici nu trebuie
    }
}

// ── Cod care lucrează prin interfața comună IWithdrawable ─────────────
// Contractul IWithdrawable NU garantează balance >= 0 — doar că Withdraw()
// scade soldul în limita permisă de fiecare implementare.
// Substituția este corectă: DebitAccount și CreditAccount respectă același contract.
public class PaymentProcessor
{
    public void ProcessPayment(IWithdrawable account, decimal amount)
    {
        // Nu știe dacă e DebitAccount sau CreditAccount — nu îi pasă
        // Dacă suma depășește limita, primește InvalidOperationException
        // — comportament documentat în contractul IWithdrawable
        account.Withdraw(amount);
        Console.WriteLine($"Payment of {amount} processed. New balance: {account.Balance}");
    }
}
✓
Ce s-a rezolvat

DebitAccount și CreditAccount sunt acum peer-uri — implementează același contract IWithdrawable, fiecare cu propriile invariante, fără să le moștenească una pe ale celeilalte. PaymentProcessor funcționează corect cu ambele. Niciun cod nu se mai poate baza pe ipoteza falsă că balance >= 0 — pentru că IWithdrawable nu face această promisiune. Dacă cineva chiar are nevoie de garanția balance >= 0, poate depinde explicit de DebitAccount sau poate cere o interfață suplimentară INonNegativeBalance.

Exemplu concret: întărirea precondiției duce la LSP încălcat

Scenariul: un sistem de procesare a plăților cu o clasă de bază PaymentGateway care acceptă orice sumă pozitivă. O subclasă PremiumGateway decide să accepte doar sume mai mari de 100 — adică întărește precondiția. Vedem pas cu pas de ce asta sparge codul existent.

PaymentGateway.cs — contractul bazei C# 
/// Procesează o plată.
/// Precondiție:  amount > 0          ← orice sumă pozitivă este validă
/// Postcondiție: returnează un TransactionId nenul
public class PaymentGateway
{
    public virtual string ProcessPayment(decimal amount)
    {
        if (amount <= 0)
            throw new ArgumentException("Amount must be positive");

        // simulăm procesarea
        return $"TXN-{Guid.NewGuid():N}";
    }
}

// ❌ PremiumGateway ÎNTĂREȘTE precondiția: acceptă doar sume > 100
// Baza promitea că amount > 0 este suficient — subclasa retrage promisiunea
public class PremiumGateway : PaymentGateway
{
    public override string ProcessPayment(decimal amount)
    {
        if (amount <= 100)           // ← precondiție mai strictă decât baza!
            throw new ArgumentException("PremiumGateway requires amount > 100");

        return $"PREMIUM-TXN-{Guid.NewGuid():N}";
    }
}

Până acum pare inofensiv — „PremiumGateway e pentru sume mari, normal că are restricții". Problema apare când codul existent, scris să lucreze cu PaymentGateway, primește în schimb un PremiumGateway:

OrderService.cs — codul care se sparge C# 
public class OrderService
{
    private readonly PaymentGateway _gateway;

    public OrderService(PaymentGateway gateway)
        => _gateway = gateway;

    public void PlaceOrder(Order order)
    {
        // OrderService cunoaște contractul bazei: amount > 0 este suficient
        // A respectat precondiția — amount = 45, deci > 0 ✓
        var txnId = _gateway.ProcessPayment(order.Total); // order.Total = 45m
        Console.WriteLine($"Order placed, transaction: {txnId}");
    }
}

// ── La configurarea aplicației ────────────────────────────────────────
var service = new OrderService(new PaymentGateway());
service.PlaceOrder(orderOf45);   // ✓ funcționează corect

// Cineva schimbă implementarea — par echivalente, nu?
var service2 = new OrderService(new PremiumGateway());
service2.PlaceOrder(orderOf45);  // ❌ ArgumentException — LSP încălcat!
//                                    OrderService a respectat contractul bazei.
//                                    Subclasa a retras o promisiune fără să anunțe.
💥
De ce e periculos — nu e doar o excepție

OrderService și-a respectat partea lui de contract: a trimis amount > 0. A respectat tot ce promitea documentația bazei. Eroarea nu e în OrderService — e în faptul că PremiumGateway a retras o promisiune pe care baza o făcuse. Orice cod scris corect față de bază se poate sparge când primește subclasa, fără nicio greșeală proprie. În sisteme mari, unde OrderService și PremiumGateway sunt în module diferite, acest bug poate fi extrem de greu de depistat.

Există trei soluții, în ordinea preferinței:

❌ Varianta greșită — verificare în apelant (anti-pattern)
// "Soluția" naivă: OrderService verifică tipul
public void PlaceOrder(Order order)
{
    if (_gateway is PremiumGateway && order.Total <= 100)
        throw new InvalidOperationException("Use standard gateway");

    var txnId = _gateway.ProcessPayment(order.Total);
}
// ❌ OrderService trebuie să cunoască subclasele — cuplare maximă
// ❌ La fiecare subclasă nouă, OrderService se modifică — OCP încălcat
// ❌ Exact comportamentul pe care LSP vrea să îl elimine
✅ Varianta corectă — contracte separate, fără moștenire forțată
// PremiumGateway nu extinde PaymentGateway — are alt contract
public interface IPaymentGateway
{
    /// Precondiție: amount > 0
    string ProcessPayment(decimal amount);
}

public interface IPremiumPaymentGateway
{
    /// Precondiție: amount > 100
    string ProcessPayment(decimal amount);
}

public class StandardGateway : IPaymentGateway
{
    public string ProcessPayment(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("amount > 0 required");
        return $"TXN-{Guid.NewGuid():N}";
    }
}

public class PremiumGateway : IPremiumPaymentGateway
{
    public string ProcessPayment(decimal amount)
    {
        if (amount <= 100) throw new ArgumentException("amount > 100 required");
        return $"PREMIUM-{Guid.NewGuid():N}";
    }
}

// OrderService depinde de IPaymentGateway — contractul cu amount > 0
// Nu poate primi PremiumGateway — compilatorul o împiedică
public class OrderService
{
    private readonly IPaymentGateway _gateway;
    public OrderService(IPaymentGateway gateway) => _gateway = gateway;

    public void PlaceOrder(Order order)
    {
        var txnId = _gateway.ProcessPayment(order.Total);
        Console.WriteLine($"Transaction: {txnId}");
    }
}
✓
Regula de aur pentru precondiții

Dacă o subclasă nu poate accepta tot ce accepta baza, nu aparține acelei ierarhii — are un contract diferit și trebuie să exprime asta printr-o interfață separată. Compilatorul devine aliatul tău: OrderService(IPaymentGateway) nu poate primi un PremiumGateway — eroarea este la compile-time, nu la runtime în producție.

Rezumat: regulile LSP prin prisma DbC

1
Precondițiile: subclasa acceptă cel puțin ce acceptă baza
Poți relaxa (accepta mai mult), nu poți întări (accepta mai puțin). Verificările de precondiție se fac o singură dată, în bază — nu se re-verifică în subclase.
2
Postcondițiile: subclasa garantează cel puțin ce garantează baza
Poți întări (garanta mai mult), nu poți slăbi (garanta mai puțin). Verificările de postcondiție se fac în bază, după apelul metodei delegate.
3
Invariantele: subclasa le păstrează pe toate ale bazei
Poți adăuga invariante noi, nu poți viola cele existente. Dacă nu poți păstra o invariantă, nu aparții ierarhiei.
4
Nicio excepție neașteptată
Subclasa nu aruncă tipuri de excepții noi, nedeclarate de bază. Codul care prinde InsufficientFundsException de la bază nu se așteaptă la NetworkException din subclasă.

05Relații între SRP, OCP și LSP

Cele trei principii nu sunt independente — se susțin reciproc:

🔗
Cum colaborează

SRP asigură că fiecare clasă are o responsabilitate clară — ceea ce face posibilă aplicarea OCP: o clasă cu o singură responsabilitate este mai ușor de „închis" pentru modificare. OCP se bazează pe polimorfism și interfețe — exact mecanismele pe care LSP le face sigure: dacă subclasele respectă contractul, substituția prin interfețe funcționează corect.

Un mod simplu de a ține minte relația: SRP spune ce trebuie să conțină o clasă, OCP spune cum trebuie să evolueze, iar LSP spune ce condiții trebuie să respecte ierarhiile de tipuri.

Codul care respectă SRP este mai ușor de extins conform OCP. Ierarhiile care respectă LSP fac OCP-ul sigur. Toate trei împreună produc cod care crește fără să se destrame.

Cerințe SOLID
SRP Cerința 1 — Separarea responsabilităților
  • UI-ul nu conține logică de business — event handlers apelează exclusiv TaskService
  • TaskService nu accesează direct baza de date — folosește ITaskRepository
  • TaskValidator este o clasă separată cu o singură responsabilitate: validarea regulilor (titlu nenul, titlu max 200 caractere, DueDate în viitor pentru DeadlineTask)
  • Justificați în README.md: care este actorul fiecărei clase și de ce o schimbare a unui actor nu afectează celelalte clase
OCP Cerința 2 — Extensibilitatea notificărilor
  • La marcarea unei sarcini ca Done, sistemul trimite o notificare al cărei tip este stocat în câmpul NotificationType
  • Definiți interfața ITaskNotifier cu metoda Notify(TaskItem task)
  • Implementați: EmailNotifier (scrie în consolă „Email sent to…"), ConsoleNotifier, FileLogNotifier (append în tasks.log)
  • TaskService primește un IReadOnlyDictionary<NotificationType, ITaskNotifier> prin constructor — selectează notifier-ul corect fără if/switch
  • Demonstrați OCP: adăugați SlackNotifier și înregistrați-l în dicționar — nicio clasă existentă nu se modifică
LSP + DbC Cerința 3 — Ierarhia de sarcini cu contract explicit
  • Subtipuri: TaskItem (bază), RecurringTask, DeadlineTask
  • Contractul metodei Complete() definit în clasa de bază: precondiție — sarcina nu este deja Done; postcondiție — Status == Done după apel; invariantă — Status nu poate fi simultan Done și Overdue
  • Baza verifică precondiția și postcondiția; subclasele suprascriu doar CompleteCore() (Template Method intern)
  • RecurringTask.CompleteCore(): setează Status = Done și recalculează DueDate += RecurrenceInterval — respectă postcondiția, adaugă comportament suplimentar
  • DeadlineTask.CompleteCore(): setează Status = Done — identic cu baza, fără surprize
REPO Cerința 4 — Repository și acces la date
  • Interfața ITaskRepository: GetAll(), GetById(int id), Add(TaskItem t), Update(TaskItem t), Delete(int id)
  • SqliteTaskRepository folosește Microsoft.Data.Sqlite. Mapare manuală din SqliteDataReader în obiect — fără ORM
  • InMemoryTaskRepository — implementare pe List<TaskItem>, folosită exclusiv în teste NUnit
  • Ambele implementări sunt interschimbabile față de TaskService — LSP respectat la nivel de repository
TESTE Cerința 5 — Teste NUnit
  • Minim 8 teste NUnit (NUnit + NUnit3TestAdapter + Microsoft.NET.Test.Sdk), toate folosind InMemoryTaskRepository — fără SQLite în teste
  • Test parametrizat ([TestCase]) care verifică postcondiția LSP: Complete() produce Status == Done pentru toate cele 3 subtipuri
  • Test care verifică că precondiția aruncă excepție la apelarea Complete() pe o sarcină deja Done
  • Test care verifică extensibilitatea OCP: un MockNotifier injectat în TaskService este apelat la finalizare, fără modificări în serviciu
  • Test care verifică că TaskValidator respinge titlul gol și titlul mai lung de 200 caractere

Nu trebuie să știți acum cum se rezolvă. Construiți proiectul conform cerințelor de mai sus, structurat și compilabil. La Laboratorul 4 rezolvăm împreună cele două probleme de mai sus pe codul vostru real.
