Design Patterns Structurale — Partea II

Ultimele patru pattern-uri structurale din catalogul GoF: Composite — structuri arborescente uniforme; Proxy — control de acces prin intermediar; Bridge — abstracție și implementare care variază independent; Flyweight — partajarea stării comune între mii de obiecte.

Obiective
  • Modelați structuri arborescente recursiv cu Composite și aplicați-l pe trei domenii diferite
  • Distingeți cele trei variante de Proxy și știți când se potrivește fiecare
  • Aplicați Bridge pentru a separa două axe de variație independente
  • Înțelegeți distincția intrinsec/extrinsec din Flyweight și când justifică complexitatea adăugată

02Composite

🌳
Composite
Compune obiectele în structuri arborescente pentru a reprezenta ierarhii parte-întreg. Permite clienților să trateze uniform obiectele individuale și compozițiile lor.
Structural GoF

Forța pattern-ului stă în faptul că clientul nu știe și nu trebuie să știe dacă lucrează cu un nod frunză sau cu un nod compus — apelează aceeași operație pe aceeași interfață, iar structura arborelui se ocupă de restul recursiv.

Structura generică (GoF)

Structura C# — participanți canonici

Composite — structura generică C# 
// Component — interfața comună pentru Leaf și Composite
public interface IComponent
{
    string Operation();
}

// Leaf — nod terminal, nu are copii
public class Leaf : IComponent
{
    private readonly string _name;
    public Leaf(string name) => _name = name;
    public string Operation() => _name;
}

// Composite — nod intern, conține alți IComponent (Leaf sau Composite)
public class Composite : IComponent
{
    private readonly string _name;
    private readonly List<IComponent> _children = new();

    public Composite(string name) => _name = name;

    public void Add(IComponent c)    => _children.Add(c);
    public void Remove(IComponent c) => _children.Remove(c);

    // Operația se propagă recursiv la toți copiii
    public string Operation()
    {
        var results = _children.Select(c => c.Operation());
        return $"{_name}({string.Join(", ", results)})";
    }
}

// Client — tratează uniform Leaf și Composite prin IComponent
IComponent tree =
    new Composite("root")
    {
        // Add() se poate apela și cu sintaxă de inițializare
    };
(Composite)tree.Add(new Leaf("A"));
(Composite)tree.Add(new Leaf("B"));
Console.WriteLine(tree.Operation()); // root(A, B)
Exemplul 1 Sistem de fișiere — GetSize() recursiv

Un director calculează dimensiunea totală prin delegare recursivă la copii. Clientul apelează GetSize() fără să știe dacă lucrează cu un fișier simplu sau o structură de directoare imbricate.

FileSystem.cs C# 
public interface IFileSystemEntry
{
    string  Name    { get; }
    long    GetSize();
    void    Print(int indent = 0);
}

// Leaf — fișier simplu
public class File : IFileSystemEntry
{
    public string Name { get; }
    private readonly long _size;

    public File(string name, long sizeBytes) => (Name, _size) = (name, sizeBytes);

    public long GetSize() => _size;
    public void Print(int indent = 0)
        => Console.WriteLine($"{new string(' ', indent * 2)}📄 {Name} ({_size} bytes)");
}

// Composite — director care poate conține fișiere sau alte directoare
public class Directory : IFileSystemEntry
{
    public string Name { get; }
    private readonly List<IFileSystemEntry> _entries = new();

    public Directory(string name) => Name = name;

    public void Add(IFileSystemEntry entry) => _entries.Add(entry);

    // Dimensiunea unui director = suma recursivă a tuturor copiilor
    public long GetSize() => _entries.Sum(e => e.GetSize());

    public void Print(int indent = 0)
    {
        Console.WriteLine($"{new string(' ', indent * 2)}📁 {Name}/ ({GetSize()} bytes)");
        foreach (var entry in _entries)
            entry.Print(indent + 1);
    }
}

// Construire arbore
var root = new Directory("proiect");
var src  = new Directory("src");
src.Add(new File("Program.cs",  4_200));
src.Add(new File("Service.cs",  8_500));
root.Add(src);
root.Add(new File("README.md", 1_200));

root.Print();
// 📁 proiect/ (13900 bytes)
//   📁 src/ (12700 bytes)
//     📄 Program.cs (4200 bytes)
//     📄 Service.cs (8500 bytes)
//   📄 README.md (1200 bytes)
Console.WriteLine($"Total: {root.GetSize()} bytes");
Exemplul 2 Expresii aritmetice — evaluare și parcurgere în trei ordini

Un arbore de expresii este cel mai natural exemplu de structură recursivă. Nodurile frunză sunt valori numerice, nodurile interne sunt operatori binari. Aceeași interfață IExpression suportă evaluarea și afișarea în cele trei ordini de parcurgere.

💡
Cele trei ordini de parcurgere

Inordine (stânga → rădăcină → dreapta): produce expresia infixată, lizibilă pentru oameni — (3 + 4) * 2. Preordine (rădăcină → stânga → dreapta): notație poloneză — operatorul precede operanzii — * + 3 4 2. Postordine (stânga → dreapta → rădăcină): notație poloneză inversă — operatorul urmează operanzilor — 3 4 + 2 *, folosită de calculatoarele cu stivă.

ExpressionTree.cs C# 
public interface IExpression
{
    double Evaluate();
    string Inorder();    // (stânga op dreapta)
    string Preorder();   // op stânga dreapta
    string Postorder(); // stânga dreapta op
}

// Leaf — valoare numerică
public class NumberExpression : IExpression
{
    private readonly double _value;
    public NumberExpression(double value) => _value = value;

    public double Evaluate()  => _value;
    public string Inorder()   => _value.ToString();
    public string Preorder()  => _value.ToString();
    public string Postorder() => _value.ToString();
}

// Composite — operator binar cu doi operanzi IExpression
public class BinaryExpression : IExpression
{
    private readonly IExpression _left, _right;
    private readonly char _op;

    public BinaryExpression(IExpression left, char op, IExpression right)
        => (_left, _op, _right) = (left, op, right);

    public double Evaluate() => _op switch
    {
        '+' => _left.Evaluate() + _right.Evaluate(),
        '-' => _left.Evaluate() - _right.Evaluate(),
        '*' => _left.Evaluate() * _right.Evaluate(),
        '/' => _right.Evaluate() == 0
                  ? throw new DivideByZeroException()
                  : _left.Evaluate() / _right.Evaluate(),
        _ => throw new NotSupportedException($"Operator necunoscut: {_op}")
    };

    // Inordine: adăugăm paranteze pentru claritate
    public string Inorder()   => $"({_left.Inorder()} {_op} {_right.Inorder()})";
    public string Preorder()  => $"{_op} {_left.Preorder()} {_right.Preorder()}";
    public string Postorder() => $"{_left.Postorder()} {_right.Postorder()} {_op}";
}

// Construim arborele pentru (3 + 4) * 2
IExpression expr = new BinaryExpression(
    new BinaryExpression(
        new NumberExpression(3),
        '+',
        new NumberExpression(4)
    ),
    '*',
    new NumberExpression(2)
);

Console.WriteLine($"Evaluate:  {expr.Evaluate()}");  // 14
Console.WriteLine($"Inorder:   {expr.Inorder()}");   // ((3 + 4) * 2)
Console.WriteLine($"Preorder:  {expr.Preorder()}");  // * + 3 4 2
Console.WriteLine($"Postorder: {expr.Postorder()}"); // 3 4 + 2 *
Exemplul 3 Forme grafice — primitive și picturi compuse

Primitivele grafice (Line, Circle, Rectangle) sunt noduri frunză. O Picture este un nod compus care poate conține orice mix de primitive și alte picturi. Operațiile Draw(), Move() și Scale() se propagă recursiv — clientul nu distinge între o primitivă și o pictură complexă.

Shapes.cs C# 
// Component — interfața comună pentru orice formă
public interface IShape
{
    void   Draw();
    void   Move(double dx, double dy);
    void   Scale(double factor);
    string Describe();
}

// ── Primitive (Leaf) ──────────────────────────────────────────────────
public class Circle : IShape
{
    public double X, Y, Radius;

    public Circle(double x, double y, double radius)
        => (X, Y, Radius) = (x, y, radius);

    public void Draw()                          => Console.WriteLine(Describe());
    public void Move(double dx, double dy)     => (X, Y) = (X + dx, Y + dy);
    public void Scale(double f)               => Radius *= f;
    public string Describe() => $"Circle  center=({X:F1},{Y:F1}) r={Radius:F1}";
}

public class Line : IShape
{
    public double X1, Y1, X2, Y2;

    public Line(double x1, double y1, double x2, double y2)
        => (X1, Y1, X2, Y2) = (x1, y1, x2, y2);

    public void Draw()                        => Console.WriteLine(Describe());
    public void Move(double dx, double dy)   => (X1, Y1, X2, Y2) = (X1+dx, Y1+dy, X2+dx, Y2+dy);
    public void Scale(double f)              => (X1, Y1, X2, Y2) = (X1*f, Y1*f, X2*f, Y2*f);
    public string Describe() => $"Line    ({X1:F1},{Y1:F1})→({X2:F1},{Y2:F1})";
}

public class Rectangle : IShape
{
    public double X, Y, Width, Height;

    public Rectangle(double x, double y, double w, double h)
        => (X, Y, Width, Height) = (x, y, w, h);

    public void Draw()                        => Console.WriteLine(Describe());
    public void Move(double dx, double dy)   => (X, Y) = (X + dx, Y + dy);
    public void Scale(double f)              => (Width, Height) = (Width*f, Height*f);
    public string Describe() => $"Rect    ({X:F1},{Y:F1}) {Width:F1}x{Height:F1}";
}

// ── Composite — grupare de forme ───────────────────────────────────────
public class Picture : IShape
{
    private readonly string _name;
    private readonly List<IShape> _shapes = new();

    public Picture(string name) => _name = name;

    public void Add(IShape s)    => _shapes.Add(s);
    public void Remove(IShape s) => _shapes.Remove(s);
    public int  Count           => _shapes.Count;

    // Toate operațiile se propagă recursiv la fiecare formă din grup
    public void Draw()
    {
        Console.WriteLine($"── Picture: {_name} ({_shapes.Count} shapes) ──");
        foreach (var s in _shapes) s.Draw();
    }

    public void Move(double dx, double dy)
        => _shapes.ForEach(s => s.Move(dx, dy));

    public void Scale(double factor)
        => _shapes.ForEach(s => s.Scale(factor));

    public string Describe() => $"Picture '{_name}' [{_shapes.Count} shapes]";
}

// ── Utilizare — clientul tratează totul ca IShape ─────────────────────
var logo = new Picture("logo");
logo.Add(new Circle(50, 50, 30));
logo.Add(new Rectangle(20, 80, 60, 20));

var scene = new Picture("scenă");
scene.Add(logo);                        // Picture în Picture
scene.Add(new Line(0, 0, 200, 0));   // primitivă alături

scene.Draw();
scene.Scale(2.0);   // scalează tot: logo + linie
scene.Move(10, 5); // mută tot cu (10, 5)
scene.Draw();      // clientul nu știe ce e în interior
✓ Când se folosește
Când structura problemei este naturalmente ierarhică și recursivă. Când clientul trebuie să trateze uniform elementele simple și colecțiile lor. Arbori de expresii, sisteme de fișiere, ierarhii UI, structuri organizaționale.
✗ Când nu se potrivește
Când componentele din ierarhie sunt fundamental diferite și nu pot partaja o interfață comună fără constrângeri artificiale. Când adâncimea arborelui este fixă și mică — o structură simplă e mai clară.

03Proxy

🛡
Proxy
Furnizează un înlocuitor sau placeholder pentru un alt obiect, pentru a controla accesul la acesta.
Structural GoF

Proxy implementează aceeași interfață ca obiectul real (Subject) și îl înlocuiește în ochii clientului. Totul trece prin Proxy — care poate amâna crearea, controla accesul, adăuga logging sau cache, înainte/după delegarea apelului.

Structura generică (GoF)

Diagramă de clase
classDiagram class Subject { <<interface>> +request() } class RealSubject { +request() } class Proxy { -realSubject RealSubject +request() } class Client Subject <|.. RealSubject Subject <|.. Proxy Proxy --> RealSubject : delegates Client ..> Subject : uses
Diagramă de secvență
sequenceDiagram participant C as Client participant P as Proxy participant R as RealSubject C->>P: request() Note over P: control acces / lazy init P->>R: request() R-->>P: rezultat Note over P: post-procesare optionala P-->>C: rezultat

Structura C# — participanți canonici

Proxy — structura generică C# 
// Subject — interfața comună pentru RealSubject și Proxy
public interface ISubject
{
    string Request();
}

// RealSubject — obiectul real, potențial costisitor de creat sau accesat
public class RealSubject : ISubject
{
    public RealSubject()
        => Console.WriteLine("RealSubject: instanțiat (operație costisitoare)");

    public string Request() => "RealSubject: rezultat";
}

// Proxy — controlează accesul la RealSubject
public class Proxy : ISubject
{
    private RealSubject? _real; // null până la primul apel (lazy)

    public string Request()
    {
        // Lazy initialization — RealSubject creat doar când e nevoie
        _real ??= new RealSubject();
        Console.WriteLine("Proxy: delegare la RealSubject");
        return _real.Request();
    }
}

// Client — lucrează cu ISubject; nu știe că e Proxy
ISubject subject = new Proxy(); // RealSubject NU e creat încă
Console.WriteLine(subject.Request()); // abia acum se instanțiază
Console.WriteLine(subject.Request()); // refolosește instanța existentă

Cele trei variante principale

Virtual Proxy Amână instanțierea costisitoare (lazy initialization)
LazyReportProxy.cs C# 
public interface IReportGenerator
{
    string Generate(string reportType);
}

// RealSubject — conectare la DB, inițializare cache: durează 2-3 secunde
public class ReportGenerator : IReportGenerator
{
    public ReportGenerator()
    {
        Console.WriteLine("ReportGenerator: inițializare (conectare DB, cache...)");
        Thread.Sleep(2000); // simulăm operație lentă
    }
    public string Generate(string type) => $"Raport [{type}] generat";
}

// Virtual Proxy — amână crearea până la primul apel Generate()
public class LazyReportProxy : IReportGenerator
{
    private ReportGenerator? _generator;

    public string Generate(string type)
    {
        _generator ??= new ReportGenerator(); // creat doar când e nevoie
        return _generator.Generate(type);
    }
}

// Aplicația pornește instant — ReportGenerator nu e creat până la primul raport
IReportGenerator gen = new LazyReportProxy();
Console.WriteLine("Aplicație pornită. Apăsați Enter pentru raport...");
Console.ReadLine(); // abia acum se inițializează ReportGenerator
Console.WriteLine(gen.Generate("lunar"));
Protection Proxy Controlează accesul pe baza permisiunilor
SecureEmployeeProxy.cs C# 
public interface IEmployeeRepository
{
    decimal GetSalary(int employeeId);
    string  GetName(int employeeId);
}

public class EmployeeRepository : IEmployeeRepository
{
    public decimal GetSalary(int id) => 5000m + id * 100m;
    public string  GetName(int id)   => $"Angajat #{id}";
}

// Protection Proxy — datele salariale sunt vizibile doar pentru rolul HR
public class SecureEmployeeProxy : IEmployeeRepository
{
    private readonly EmployeeRepository _repo = new();
    private readonly string _currentUserRole;

    public SecureEmployeeProxy(string role) => _currentUserRole = role;

    public decimal GetSalary(int id)
    {
        if (_currentUserRole != "HR")
            throw new UnauthorizedAccessException(
                "Accesul la date salariale este permis doar rolului HR.");
        return _repo.GetSalary(id);
    }

    // GetName() este accesibil oricui — nicio restricție
    public string GetName(int id) => _repo.GetName(id);
}

IEmployeeRepository hrView  = new SecureEmployeeProxy("HR");
IEmployeeRepository devView = new SecureEmployeeProxy("Developer");

Console.WriteLine(hrView.GetSalary(1));  // 5100 — OK
Console.WriteLine(devView.GetName(1));  // "Angajat #1" — OK
devView.GetSalary(1);                    // UnauthorizedAccessException
Remote Proxy Reprezintă local un obiect aflat la distanță

Remote Proxy oferă clientului o interfață locală pentru un serviciu aflat pe altă mașină sau proces. Clientul apelează metode ca și cum ar lucra local — Proxy-ul se ocupă de comunicarea în rețea, serializare și tratarea erorilor de conexiune.

RemoteWeatherProxy.cs C# 
public interface IWeatherService
{
    string GetForecast(string city);
}

// RealSubject — serviciu real, rulează pe alt server
// Clientul nu îl poate instanția direct
public class WeatherService : IWeatherService
{
    public string GetForecast(string city) => $"Sunny, 22°C in {city}";
}

// Remote Proxy — simulează un apel HTTP către serviciul extern
public class RemoteWeatherProxy : IWeatherService
{
    private readonly string _baseUrl;
    private readonly HttpClient _http = new();

    public RemoteWeatherProxy(string baseUrl) => _baseUrl = baseUrl;

    public string GetForecast(string city)
    {
        try
        {
            // Clientul face un apel local — Proxy traduce în HTTP
            var url      = $"{_baseUrl}/forecast?city={Uri.EscapeDataString(city)}";
            var response = _http.GetStringAsync(url).GetAwaiter().GetResult();
            return response;
        }
        catch (HttpRequestException ex)
        {
            // Proxy gestionează erorile de rețea — clientul nu le vede
            return $"[Offline] Date indisponibile: {ex.Message}";
        }
    }
}

// Client — apelează IWeatherService fără să știe că e un serviciu distant
IWeatherService weather = new RemoteWeatherProxy("https://api.weather.example.com");
Console.WriteLine(weather.GetForecast("București"));
Caching Proxy Reține rezultatele costisitoare și le returnează din cache

Caching Proxy interceptează apelurile și returnează rezultatul stocat dacă există deja. Apelul real către RealSubject se face doar la prima cerere sau când cache-ul expiră. Clientul nu știe că lucrează cu un cache — interfața este identică.

CachingWeatherProxy.cs C# 
// Caching Proxy — stochează rezultatele pentru un interval de timp
public class CachingWeatherProxy : IWeatherService
{
    private readonly IWeatherService _inner;
    private readonly TimeSpan         _ttl;     // durata de viață a intrărilor

    private readonly Dictionary<string, (string Result, DateTime CachedAt)> _cache = new();

    public CachingWeatherProxy(IWeatherService inner, TimeSpan ttl)
        => (_inner, _ttl) = (inner, ttl);

    public string GetForecast(string city)
    {
        if (_cache.TryGetValue(city, out var entry) &&
            DateTime.UtcNow - entry.CachedAt < _ttl)
        {
            Console.WriteLine($"[CACHE HIT] {city}");
            return entry.Result; // returnăm din cache, fără apel real
        }

        Console.WriteLine($"[CACHE MISS] {city} — apel real");
        var result = _inner.GetForecast(city);
        _cache[city] = (result, DateTime.UtcNow);
        return result;
    }

    public int  CacheSize  => _cache.Count;
    public void Invalidate(string city) => _cache.Remove(city);
    public void InvalidateAll()          => _cache.Clear();
}

// Combinare: Remote Proxy + Caching Proxy în lanț
IWeatherService weather =
    new CachingWeatherProxy(
        new RemoteWeatherProxy("https://api.weather.example.com"),
        ttl: TimeSpan.FromMinutes(10)
    );

weather.GetForecast("București"); // CACHE MISS — apel HTTP
weather.GetForecast("București"); // CACHE HIT  — din memorie
weather.GetForecast("Cluj");       // CACHE MISS — apel HTTP
weather.GetForecast("Cluj");       // CACHE HIT  — din memorie
💡
Caching Proxy vs Decorator

Un Caching Proxy arată identic structural cu un Decorator — amândouă învelesc un obiect prin aceeași interfață. Distincția este din nou de intenție: Caching Proxy controlează când se apelează RealSubject (îl evită când are deja răspunsul), pe când Decorator adaugă comportament la fiecare apel. Când cache-ul are intrarea, RealSubject nu mai este apelat deloc — lucru imposibil într-un Decorator pur.

🧭
Proxy vs Decorator

Proxy controlează accesul la obiectul real — de obicei gestionează și ciclul de viață al acestuia (lazy init, distrugere). Decorator adaugă comportament unui obiect deja existent și accesibil — nu se ocupă de ciclul de viață. Structural sunt identice; intenția le diferențiază.

✓ Cele patru variante
Virtual Proxy — obiect costisitor de creat, folosit rar (lazy init). Protection Proxy — control de acces pe baza rolurilor sau permisiunilor. Remote Proxy — obiect aflat pe alt server sau proces, accesat transparent. Caching Proxy — rezultate costisitoare, cerute repetat în interval scurt.
✗ Când nu se potrivește
Când adăugarea unui nivel de indirectare produce latență inacceptabilă pe căi critice de performanță. Dacă scopul este adăugarea de comportament, nu controlul accesului — Decorator e mai potrivit.

04Bridge

🌉
Bridge
Decuplează o abstracție de implementarea sa, astfel încât cele două să poată varia independent.
Structural GoF

Când o clasă are două axe de variație independente, moștenirea produce o explozie combinatorială. Bridge separă cele două dimensiuni în ierarhii distincte conectate printr-o referință (bridge-ul). Fiecare poate evolua separat, fără să afecteze cealaltă.

Structura generică (GoF)

Diagramă de clase

ConcreteImplementorA Implementor <|.. ConcreteImplementorB
Diagramă de secvență
sequenceDiagram participant C as Client participant A as Abstraction participant I as Implementor C->>A: operation() A->>I: operationImpl() I-->>A: rezultat A-->>C: rezultat procesat

Structura C# — participanți canonici

Bridge — structura generică C# 
// Implementor — interfața pentru axa de implementare
public interface IImplementor
{
    string OperationImpl();
}

public class ConcreteImplementorA : IImplementor
{
    public string OperationImpl() => "ImplementorA";
}

public class ConcreteImplementorB : IImplementor
{
    public string OperationImpl() => "ImplementorB";
}

// Abstraction — axa de abstracție, conține implementorul prin compoziție
public abstract class Abstraction
{
    protected readonly IImplementor _impl;

    protected Abstraction(IImplementor impl) => _impl = impl;

    public virtual string Operation()
        => $"Abstraction + {_impl.OperationImpl()}";
}

// RefinedAbstraction — extinde abstracția fără să știe de implementări concrete
public class RefinedAbstraction : Abstraction
{
    public RefinedAbstraction(IImplementor impl) : base(impl) {}

    public override string Operation()
        => $"RefinedAbstraction + {_impl.OperationImpl()}";
}

// Combinare liberă: oricare Abstraction × oricare Implementor
Abstraction ab1 = new RefinedAbstraction(new ConcreteImplementorA());
Abstraction ab2 = new RefinedAbstraction(new ConcreteImplementorB());
Console.WriteLine(ab1.Operation()); // RefinedAbstraction + ImplementorA
Console.WriteLine(ab2.Operation()); // RefinedAbstraction + ImplementorB

Exemplu — sistem de notificări cu două axe

Avem două dimensiuni de variație independente: tipul de notificare (urgentă, programată) și canalul de livrare (email, SMS). Prin moștenire: 2 × 2 = 4 clase. Cu Bridge: 2 + 2 + interfața = mai puțin, și fiecare axă poate crește independent.

Notifications.cs C# 
// Implementor — canalul de livrare
public interface IMessageSender
{
    void Send(string recipient, string message);
}

public class EmailSender : IMessageSender
{
    public void Send(string to, string msg)
        => Console.WriteLine($"[EMAIL → {to}] {msg}");
}

public class SmsSender : IMessageSender
{
    public void Send(string to, string msg)
        => Console.WriteLine($"[SMS → {to}] {msg}");
}

// Abstraction — tipul de notificare, conține IMessageSender
public abstract class Notification
{
    protected readonly IMessageSender _sender;
    protected Notification(IMessageSender sender) => _sender = sender;
    public abstract void Notify(string recipient, string message);
}

// RefinedAbstraction 1 — notificare urgentă
public class UrgentNotification : Notification
{
    public UrgentNotification(IMessageSender sender) : base(sender) {}

    public override void Notify(string recipient, string message)
        => _sender.Send(recipient, $"[URGENT] {message}");
}

// RefinedAbstraction 2 — notificare programată
public class ScheduledNotification : Notification
{
    private readonly DateTime _scheduledAt;

    public ScheduledNotification(IMessageSender sender, DateTime at)
        : base(sender) => _scheduledAt = at;

    public override void Notify(string recipient, string message)
        => _sender.Send(recipient, $"[La {_scheduledAt:HH:mm}] {message}");
}

// Combinare liberă: oricare tip × oricare canal
Notification n1 = new UrgentNotification(new EmailSender());
Notification n2 = new UrgentNotification(new SmsSender());
Notification n3 = new ScheduledNotification(new EmailSender(), DateTime.Now.AddHours(2));

n1.Notify("ana@x.com", "Server down!"); // [EMAIL → ana@x.com] [URGENT] Server down!
n2.Notify("+40700000", "Server down!");  // [SMS → +40700000] [URGENT] Server down!
n3.Notify("ion@x.com", "Raport lunar");  // [EMAIL → ion@x.com] [La 14:30] Raport lunar
🧭
Bridge vs Adapter

Adapter reconciliază interfețe incompatibile existente — este o soluție aplicată retroactiv. Bridge separă deliberat abstracția de implementare din faza de design, anticipând că ambele vor varia independent. Adapter rezolvă o problemă de compatibilitate; Bridge previne o problemă de rigiditate.

✓ Când se folosește
Când o clasă are două axe de variație independente care ar produce explozie combinatorială prin moștenire. Când vreți să schimbați implementarea la runtime. Când abstracția și implementarea trebuie să evolueze separat în echipe diferite.
✗ Când nu se potrivește
Când există o singură axă de variație — simpla moștenire sau compoziție e suficientă. Bridge adaugă un nivel de indirectare care merită justificat prin numărul real de combinații necesare.

05Flyweight

🪶
Flyweight
Folosește partajarea pentru a suporta eficient un număr mare de obiecte fine-grained. Separă starea partajată (intrinsecă) de starea specifică fiecărei instanțe (extrinsecă).
Structural GoF

Conceptul central: un obiect Flyweight conține starea intrinsecă — datele care nu se schimbă între instanțe și pot fi partajate (font, dimensiune, culoare). Starea extrinsecă — datele specifice fiecărui context de utilizare (poziția unui caracter pe ecran) — este furnizată de client la fiecare apel, nu stocată în Flyweight.

Structura C# — participanți canonici

Flyweight — structura generică C# 
// Flyweight — conține starea intrinsecă partajată
public class ConcreteFlyweight
{
    // Stare intrinsecă — imutabilă, partajată între toate utilizările
    public string SharedState { get; }

    public ConcreteFlyweight(string sharedState) => SharedState = sharedState;

    // Starea extrinsecă este furnizată la fiecare apel — nu e stocată
    public void Operation(string uniqueState)
        => Console.WriteLine($"Flyweight({SharedState}) + context({uniqueState})");
}

// FlyweightFactory — pool de instanțe, creează sau returnează existente
public class FlyweightFactory
{
    private readonly Dictionary<string, ConcreteFlyweight> _pool = new();

    public ConcreteFlyweight GetFlyweight(string key)
    {
        if (!_pool.ContainsKey(key))
        {
            Console.WriteLine($"FlyweightFactory: creare nouă instanță pentru '{key}'");
            _pool[key] = new ConcreteFlyweight(key);
        }
        return _pool[key];
    }

    public int PoolSize => _pool.Count;
}

// Client furnizează starea extrinsecă la fiecare apel
var factory = new FlyweightFactory();

var fw1 = factory.GetFlyweight("shared-A"); // creare nouă
var fw2 = factory.GetFlyweight("shared-A"); // returnează aceeași instanță
var fw3 = factory.GetFlyweight("shared-B"); // creare nouă

fw1.Operation("context-1");
fw2.Operation("context-2"); // același Flyweight, context diferit
fw3.Operation("context-3");

Console.WriteLine($"Instanțe în pool: {factory.PoolSize}"); // 2, nu 3

Exemplu — editor de text cu milioane de caractere

TextEditor.cs C# 
// Flyweight — starea intrinsecă: font, dimensiune, culoare (partajate)
public class CharacterStyle
{
    public string Font  { get; }
    public int    Size  { get; }
    public string Color { get; }

    public CharacterStyle(string font, int size, string color)
        => (Font, Size, Color) = (font, size, color);

    // Starea extrinsecă (char, x, y) furnizată la randare — nu stocată
    public void Draw(char character, int x, int y)
        => Console.WriteLine($"'{character}' [{Font} {Size}pt {Color}] la ({x},{y})");
}

// Factory — pool de stiluri; un milion de caractere pot folosi 3-4 stiluri
public class CharacterStyleFactory
{
    private readonly Dictionary<string, CharacterStyle> _pool = new();

    public CharacterStyle GetStyle(string font, int size, string color)
    {
        var key = $"{font}-{size}-{color}";
        if (!_pool.TryGetValue(key, out var style))
        {
            style = new CharacterStyle(font, size, color);
            _pool[key] = style;
        }
        return style;
    }

    public int UniqueStyles => _pool.Count;
}

// Context — un caracter din document: referință la stil + poziție
public record CharacterContext(char Char, int X, int Y, CharacterStyle Style)
{
    public void Draw() => Style.Draw(Char, X, Y);
}

// Utilizare — un milion de caractere, câteva stiluri
var styleFactory = new CharacterStyleFactory();
var normalStyle  = styleFactory.GetStyle("Arial", 12, "black");
var boldStyle    = styleFactory.GetStyle("Arial", 12, "black"); // aceeași instanță!

var document = new List<CharacterContext>
{
    new('H', 0,  0,  normalStyle),
    new('e', 10, 0,  normalStyle),
    new('l', 18, 0,  normalStyle),
    // ... un milion de caractere, toate referind normalStyle
};

Console.WriteLine($"Stiluri unice în memorie: {styleFactory.UniqueStyles}"); // 1
Console.WriteLine($"Caractere în document:   {document.Count}");
document.ForEach(c => c.Draw());
⚠️
Când Flyweight nu merită

Flyweight adaugă complexitate semnificativă: separarea stării intrinsece de cea extrinsecă, factory-ul cu pool, transmiterea stării extrinsece la fiecare apel. Se justifică doar când numărul de obiecte este cu adevărat mare (zeci de mii sau mai mult) și memoria este o constrângere reală. Pentru câteva sute de obiecte, costul complexității depășește beneficiul.

✓ Când se justifică
Aplicația creează un număr foarte mare de obiecte similare (>10.000). Obiectele consumă multă memorie. Starea lor poate fi separată clar în intrinsecă (partajată) și extrinsecă (furnizată de context). Editoare grafice, motoare de jocuri, parsere de texte mari.
✗ Când nu se potrivește
Când numărul de obiecte este mic sau moderat. Când starea nu se poate separa natural în intrinsecă și extrinsecă. Flyweight face codul mai greu de înțeles și depanat — nu aplicați prematur.

06Comparație — toate 7 pattern-uri structurale

PatternProblema rezolvatăCuvânt cheieRelație SOLID
Adapter Interfețe incompatibile — integrează fără modificare traducere OCP, DIP
Decorator Extinde comportamentul la runtime fără subclassing wrapping dinamic OCP, SRP
Facade Simplifică accesul la un subsistem complex simplificare SRP, DIP
Composite Tratează uniform primitive și compozițiile lor recursivitate OCP, LSP
Proxy Controlează accesul la un obiect prin intermediar control acces SRP, OCP
Bridge Separă abstracția de implementare — variație independentă decuplare OCP, DIP
Flyweight Partajează starea comună între mii de obiecte partajare SRP
Toate pattern-urile structurale rezolvă probleme de compoziție — cum asamblezi obiectele. Diferența stă în ce problemă de compoziție rezolvă fiecare: compatibilitate, extensibilitate, simplitate, uniformitate, control, independență sau eficiență.

DrawingTool — instrument grafic de desenare

Laborator 7 — Composite + Bridge + Proxy pe un model de forme grafice

Sistemul permite construirea de scene grafice din primitive (Line, Circle, Rectangle, Ellipse) și picturi (Picture) care sunt grupuri de alte forme. Orice Picture poate conține alte picturi — ierarhie nelimitată. Operațiile Draw(), Move() și Scale() se propagă uniform prin întregul arbore.

Cerințe
COMPOSITE Cerința 1 — Ierarhia de forme
  • Interfața IShape cu metodele: Draw(ICanvas), Move(double dx, double dy), Scale(double factor), GetBoundingBox(): BoundingBox
  • Primitive (Leaf): Line, Circle, Rectangle, Ellipse — fiecare cu coordonate proprii și implementare directă a metodelor
  • Picture (Composite): conține o listă de IShape, cu metode Add(IShape) și Remove(IShape); Move() și Scale() se propagă la toți copiii; GetBoundingBox() returnează bounding box-ul unificat al tuturor copiilor
  • O Picture poate conține alte Picture — demonstrați cu o scenă imbricată pe cel puțin două niveluri
BRIDGE Cerința 2 — Separarea randării de model
  • Interfața ICanvas (Implementor): DrawLine(x1,y1,x2,y2), DrawCircle(cx,cy,r), DrawRect(x,y,w,h), DrawEllipse(cx,cy,rx,ry)
  • ConsoleCanvas: afișează fiecare operație ca text în consolă — util pentru depanare
  • SvgCanvas: acumulează elementele SVG și expune metoda GetSvg(): string care returnează un document SVG complet — util pentru export
  • Draw(ICanvas canvas) pe orice IShape folosește canvas-ul primit — modelul nu știe cum se randează
PROXY Cerința 3 — Forme blocate (read-only)
  • Implementați ReadOnlyShapeProxy : IShape care învelește orice IShape
  • Draw(canvas) — delegă normal la shape-ul interior
  • Move() și Scale() — aruncă InvalidOperationException("Shape is locked")
  • GetBoundingBox() — delegă normal
  • Demonstrați: un Circle învelit în ReadOnlyShapeProxy se poate desena dar nu poate fi mutat sau scalat
TESTE Cerința 4 — Teste NUnit
  • Test: Picture cu 3 primitive, Scale(2.0) → toate dimensiunile s-au dublat (Composite — propagare recursivă)
  • Test: GetBoundingBox() pe o Picture cu forme la coordonate cunoscute returnează bounding box-ul corect
  • Test: SvgCanvas după Draw() pe un Circle conține tagul <circle> în SVG-ul generat (Bridge)
  • Test: ReadOnlyShapeProxy.Move() aruncă InvalidOperationException (Proxy)
  • Test: ReadOnlyShapeProxy.Draw() nu aruncă excepție și delegă la shape-ul interior (Proxy — operație permisă)

💬 Teme de Discuție
  1. Composite presupune că toate nodurile din arbore au aceeași interfață. Ce faceți când un nod compus trebuie să expună operații suplimentare față de nodurile frunză (ex: Add(), Remove()) — le includeți în interfața comună sau nu?
  2. Proxy și Decorator sunt structural identice. Dați un exemplu concret din practică unde aceeași clasă ar putea fi interpretată ca ambele — și explicați de ce intenția contează mai mult decât structura.
  3. Bridge introduce un nivel de indirectare care poate afecta performanța. Cum decideți dacă flexibilitatea adusă de Bridge justifică acest cost în contextul unui sistem cu cerințe stricte de latență?
  4. Flyweight funcționează corect doar dacă starea intrinsecă este cu adevărat imutabilă. Ce se întâmplă dacă un client modifică un obiect Flyweight primit din pool? Cum preveniți această problemă în C#?
