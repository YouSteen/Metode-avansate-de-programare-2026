Principiile SOLID — Partea II

ISP și DIP completează tabloul SOLID. ISP spune că interfețele trebuie să fie mici și precise — nu interfețe supraîncărcate cu zeci de metode. DIP spune că dependențele trebuie să fie orientate spre abstracții, nu spre implementări concrete. La final, trei heuristici practice: DRY, KISS, YAGNI.

02ISP — Interface Segregation Principle

I
Interface Segregation Principle
Niciun client nu trebuie să fie forțat să depindă de metode pe care nu le folosește.

O interfață mare obligă orice clasă care o implementează să definească toate metodele, chiar și pe cele irelevante pentru ea. Și orice client care depinde de interfață este forțat să „cunoască" și să fie recompilat dacă oricare din cele zeci de metode se schimbă — inclusiv cele pe care nu le folosea deloc.

ISP spune: împarte interfața în roluri mici și precise, fiecare acoperind un rol precis. Clienții depind doar de rolul de care au nevoie.

Problema: interfața supraîncărcată

Scenariul: un sistem de raportare cu o interfață IDocument care reunește toate operațiile posibile — citire, scriere, export, previzualizare, compresie. Un client care vrea doar să citească un document este forțat să depindă de Compress() și Export(), metode pe care nu le va apela niciodată.

❌ Interfață supraîncărcată — forțează dependențe inutile
IDocument.cs — toate operațiile la un loc C# 
public interface IDocument
{
    string  GetContent();
    void    SetContent(string content);
    byte[]  Export(string format);
    void    Compress();
    string  Preview(int maxChars);
}

// ReadOnlyDocument vrea să ofere conținut, nu să îl modifice sau exporte
public class ReadOnlyDocument : IDocument
{
    private readonly string _content;
    public  string GetContent() => _content;
    public  string Preview(int n) => _content[..Math.Min(n, _content.Length)];

    // ❌ Forțat să implementeze operații care nu au sens
    public void   SetContent(string c) => throw new NotSupportedException();
    public byte[] Export(string f)    => throw new NotSupportedException();
    public void   Compress()          => throw new NotSupportedException();
}
// ISP și LSP se susțin reciproc:
// o interfață prea mare forțează implementări false → LSP încălcat.
✅ Interfețe segregate — fiecare client depinde doar de rolul său
Interfețe segregate pe roluri C# 
// Fiecare interfață acoperă un singur rol
public interface IReadableDocument
{
    string GetContent();
    string Preview(int maxChars);
}

public interface IWritableDocument
{
    void SetContent(string content);
}

public interface IExportableDocument
{
    byte[] Export(string format);
    void   Compress();
}

// ReadOnlyDocument implementează doar ce poate face — fără excepții false
public class ReadOnlyDocument : IReadableDocument
{
    private readonly string _content;
    public string GetContent()    => _content;
    public string Preview(int n) => _content[..Math.Min(n, _content.Length)];
}

// EditableDocument implementează citire + scriere
public class EditableDocument : IReadableDocument, IWritableDocument
{
    private string _content = string.Empty;
    public string GetContent()             => _content;
    public string Preview(int n)           => _content[..Math.Min(n, _content.Length)];
    public void   SetContent(string value) => _content = value;
}

// DocumentViewer depinde DOAR de IReadableDocument
// Nu poate modifica sau exporta — protecție prin design
public class DocumentViewer
{
    public void Show(IReadableDocument doc)
        => Console.WriteLine(doc.Preview(200));
}

ISP aplicat la repository — exemplu practic

Un exemplu mai aproape de practica zilnică: interfața ITaskRepository din proiectul vostru are 5 metode. Un ReportService care generează rapoarte are nevoie doar să citească date — de ce să depindă de Add, Update, Delete?

ITaskRepository.cs — împărțit în două roluri C# 
// Rolul de citire — tot ce are nevoie un raport sau o pagină de listare
public interface ITaskReader
{
    IReadOnlyList<TaskItem> GetAll();
    TaskItem?               GetById(int id);
}

// Rolul de scriere — creare, modificare, ștergere
public interface ITaskWriter
{
    void Add   (TaskItem task);
    void Update(TaskItem task);
    void Delete(int id);
}

// Interfața completă — compusă din cele două roluri (ISP + segregare)
public interface ITaskRepository : ITaskReader, ITaskWriter { }

// SqliteTaskRepository implementează totul — nicio schimbare față de înainte
public class SqliteTaskRepository : ITaskRepository
{
    // ... implementare completă
}

// ReportService depinde DOAR de ITaskReader — nu poate șterge date accidental
public class ReportService
{
    private readonly ITaskReader _reader;
    public ReportService(ITaskReader reader) => _reader = reader;

    public string GenerateSummary()
    {
        var tasks = _reader.GetAll();
        return $"Total: {tasks.Count}, Done: {tasks.Count(t => t.Status == Status.Done)}";
    }
}
✓
Ce câștigăm concret

ReportService nu poate fi modificat să apeleze Delete() din greșeală — compilatorul nu îi permite. Dacă adăugăm o metodă nouă în ITaskWriter, ReportService nu se recompilează. Testele pentru ReportService folosesc un mock simplu de ITaskReader — nu trebuie să implementeze și metodele de scriere.

Cât de mici trebuie să fie interfețele?

ISP nu spune „o metodă per interfață". Spune că interfața trebuie să reflecte un rol coerent al unui client. Două metode care sunt mereu folosite împreună aparțin aceleiași interfețe. Regula practică:

1
Identifică clienții — nu implementările
Întreabă: cine folosește interfața? Ce metode are nevoie fiecare client? Separă după client, nu după logică internă.
2
Metodele mereu folosite împreună rămân împreună
GetAll() și GetById() sunt mereu cerute de același client (orice view sau raport). Nu le separa forțat.
3
O metodă care apare doar la un singur client → interfață separată
Dacă GetStatistics() e cerută doar de un dashboard, nu de alte 10 componente, extrage-o în ITaskStatistics.
4
Compune interfețele mari din cele mici
ITaskRepository : ITaskReader, ITaskWriter — implementarea concretă poate implementa totul, dar clienții depind de bucăți.
⚠️
ISP prematur — un anti-pattern

Nu sparge interfețele înainte să știi cine le folosește. O interfață cu 3 metode pe care toți clienții le folosesc în totalitate nu necesită segregare. Aplici ISP când observi că un client e forțat să depindă de metode pe care nu le cheamă niciodată.

03DIP — Dependency Inversion Principle

D
Dependency Inversion Principle
Modulele de nivel înalt nu trebuie să depindă de modulele de nivel scăzut. Ambele trebuie să depindă de abstracții.

DIP are două afirmații. Prima: logica de business (nivel înalt) nu trebuie să depindă direct de SQLite, de HTTP, de sistemul de fișiere (nivel scăzut) — trebuie să depindă de o interfață. A doua afirmație, mai subtilă: interfața nu aparține modulului de nivel scăzut, ci modulului de nivel înalt care o definește. Asta înseamnă că dependența de compilare este inversată față de dependența de execuție.

Dependența directă — problema

❌ Nivel înalt depinde de nivel scăzut
TaskService.cs — cuplat la SQLite C# 
public class TaskService
{
    // ❌ TaskService știe că există SqliteTaskRepository
    // ❌ Nu poți testa TaskService fără o bază de date SQLite
    // ❌ Vrei să schimbi la PostgreSQL? Modifici TaskService.
    private readonly SqliteTaskRepository _repo
        = new SqliteTaskRepository("tasks.db");

    public void CompleteTask(int id)
    {
        var task = _repo.GetById(id);
        task.Complete();
        _repo.Update(task);
    }
}

// Grafic de dependențe:
// TaskService ──→ SqliteTaskRepository ──→ SQLite
// Nivel înalt      Nivel scăzut             Infrastructură
✅ Ambele depind de abstracție
TaskService.cs — cuplat doar la interfață C# 
public class TaskService
{
    // ✓ TaskService știe doar de ITaskRepository
    // ✓ Se testează cu InMemoryTaskRepository
    // ✓ Schimbi la PostgreSQL fără să atingi TaskService
    private readonly ITaskRepository _repo;

    public TaskService(ITaskRepository repo) => _repo = repo;

    public void CompleteTask(int id)
    {
        var task = _repo.GetById(id);
        task.Complete();
        _repo.Update(task);
    }
}

// Grafic de dependențe:
// TaskService ──→ ITaskRepository ←── SqliteTaskRepository
// Nivel înalt      Abstracție          Nivel scăzut
// Săgeata spre SqliteTaskRepository este INVERSATĂ față de înainte.

Dependency Inversion vs Dependency Injection

Sunt două concepte distincte, frecvent confundate:

P
Dependency Inversion (DIP) — principiu de design
Spune ce trebuie să se întâmple: modulele de nivel înalt nu depind de implementări concrete. Este o regulă arhitecturală, nu o tehnică de cod.
T
Dependency Injection (DI) — tehnică de implementare
Spune cum se poate obține DIP în practică: dependențele se furnizează din exterior (prin constructor, proprietate sau metodă) în loc să fie create intern cu new. DI implementează DIP, dar DIP se poate obține și altfel (factory, service locator).
C
IoC Container — unealtă opțională pentru DI
Un container (Microsoft.Extensions.DependencyInjection, Autofac etc.) automatizează construirea și injectarea dependențelor. Nu e obligatoriu pentru DIP — constructor injection manual este suficient pentru proiecte mici.

Constructor Injection manual — fără container

Fără un container IoC, asamblarea dependențelor se face în Composition Root — un singur loc în aplicație, de obicei Program.cs. Toate obiectele se construiesc o singură dată, la pornire.

Program.cs — Composition Root manual C# 
// Composition Root: singurul loc unde există `new` pentru dependențe
// Tot restul codului primește interfețe — nu știe ce implementare există

// 1. Construim nivelul scăzut (infrastructura)
var repository = new SqliteTaskRepository("tasks.db");

// 2. Construim notificatorii
var notifiers = new Dictionary<NotificationType, ITaskNotifier>
{
    [NotificationType.Email]    = new EmailNotifier(),
    [NotificationType.Console]  = new ConsoleNotifier(),
    [NotificationType.FileLog]  = new FileLogNotifier("tasks.log"),
};

// 3. Construim serviciile (nivel înalt) — primesc interfețe
var validator = new TaskValidator();
var service   = new TaskService(repository, validator, notifiers);
var reporter  = new ReportService(repository); // ITaskReader — ISP aplicat

// 4. Construim UI-ul — primește serviciile
var form = new MainForm(service, reporter);
Application.Run(form);

// Observați: TaskService nu știe de SqliteTaskRepository.
// SqliteTaskRepository nu știe de TaskService.
// Ambele depind doar de ITaskRepository — definit în Core.

Cu container IoC — Microsoft.Extensions.DependencyInjection

Același rezultat, dar containerul construiește și injectează automat. Containerul devine util când numărul de servicii crește și construirea manuală devine laborioasă.

Program.cs — cu container IoC C# 
var services = new ServiceCollection();

// Înregistrăm: „când cineva cere ITaskRepository, dă SqliteTaskRepository"
services.AddSingleton<ITaskRepository, SqliteTaskRepository>();
services.AddSingleton<ITaskReader(sp => sp.GetRequiredService<SqliteTaskRepository>());
services.AddTransient<TaskValidator>();
services.AddTransient<TaskService>();   // containerul injectează automat
services.AddTransient<ReportService>();

var provider = services.BuildServiceProvider();

// Containerul construiește MainForm și toate dependențele sale în lanț
var form = provider.GetRequiredService<MainForm>();
Application.Run(form);

// Lifetimes:
// AddSingleton  — o singură instanță pe toată durata aplicației
// AddTransient  — instanță nouă la fiecare cerere
// AddScoped     — o instanță per „scope" (per request în web, manual altundeva)
💡
Unde „trăiește" interfața — detaliul care face DIP real

Interfața ITaskRepository trebuie să fie definită în proiectul Core (nivel înalt), nu în proiectul Data (nivel scăzut). Astfel, proiectul Core nu depinde de Data — dimpotrivă, Data depinde de Core (implementează interfața definită acolo). Dependența de compilare este inversată față de dependența de execuție.

Testabilitate — beneficiul imediat al DIP

TaskServiceTests.cs — fără SQLite, fără fișiere C# 
[TestFixture]
public class TaskServiceTests

04DRY, KISS, YAGNI

Aceste trei heuristici nu sunt principii de design la același nivel cu SOLID — sunt reguli practice de bună conduită care ghidează deciziile de zi cu zi. Le prezentăm împreună cu SOLID pentru că interacționează direct cu el: DRY justifică de ce extrageți o abstracție, YAGNI justifică de ce nu o extragi, KISS reamintește că simplitatea este o valoare.

DRY Don't Repeat Yourself

Fiecare element de cunoaștere trebuie să aibă o singură reprezentare autorizată în sistem. Nu e vorba doar de cod duplicat — e vorba de logică duplicată. Dacă o regulă de business apare în două locuri, modificarea ei va fi uitată în unul din ele.

DRY se aplică la logică, nu la sintaxă. Două funcții care se întâmplă să arate similar dar fac lucruri diferite nu sunt duplicat DRY.

KISS Keep It Simple, Stupid

Soluția cea mai simplă care funcționează corect este de obicei cea mai bună. Complexitatea are un cost: e mai greu de înțeles, de testat, de modificat.

KISS nu înseamnă să nu refactorizezi. Înseamnă să nu adaugi straturi de abstracție care nu rezolvă o problemă reală prezentă. Dacă poți explica ce face codul în 10 cuvinte, e simplu.

YAGNI You Aren't Gonna Need It

Nu implementa funcționalitate pe care o crezi că o vei avea nevoie în viitor. Implementați doar ce aveți nevoie în prezent. Codul nefolosit are un cost: trebuie testat, menținut, citit și creează dependențe inutile.

YAGNI intră în tensiune cu OCP — dar soluția e simplă: OCP se aplică atunci când schimbarea este concretă și iminentă, nu ipotetică. Dacă nu știi că va veni, nu te pregătești pentru ea.

Cum interacționează cele trei

Exemple concrete de tensiuni C# 
// ── DRY vs YAGNI ───────────────────────────────────────────────────────
// Ai două metode cu logică similară. DRY spune: extrage abstracția.
// YAGNI spune: extrage-o doar dacă e cu adevărat aceeași regulă, nu doar cod similar.

// ❌ DRY aplicat greșit — unificare forțată a lucruri diferite
private decimal CalculateDiscount(Order order) => order.Total * 0.1m;
private decimal CalculateTax(Order order)      => order.Total * 0.1m;
// Ambele înmulțesc cu 0.1 — dar sunt reguli de business DIFERITE.
// Dacă taxa se schimbă la 0.19, discountul rămâne 0.1.
// Unificarea lor ar fi o aplicare greșită a DRY — ar crea cuplare artificială.

// ── KISS vs SOLID ──────────────────────────────────────────────────────
// Ai o singură sursă de notificare — email. KISS spune: nu crea INotifier.
// OCP spune: vei adăuga Slack în curând, creează interfața acum.
// Rezolvare: dacă Slack este o cerință confirmată pentru săptămâna viitoare → creează interfața.
// Dacă Slack rămâne ipotetică → YAGNI prevalează, interfața se adaugă când cerința devine reală.

// ── DRY bine aplicat ──────────────────────────────────────────────────
public class TaskValidator
{
    // Regula „titlul nu poate fi gol" apare la Add și la Update.
    // E aceeași regulă de business — DRY justifică extragerea.
    public void Validate(TaskItem task)
    {
        if (string.IsNullOrWhiteSpace(task.Title))
            throw new ValidationException("Title is required");
        if (task.Title.Length > 200)
            throw new ValidationException("Title max 200 chars");
    }
    // Add și Update apelează Validate() — un singur loc pentru regula de business.
}
🧭
Regula de decizie practică

Când simți nevoia să „pregătești" codul pentru viitor, întreabă: „este aceasta o cerință confirmată sau o presupunere?" Dacă e presupunere → YAGNI. Dacă aceeași logică apare în două locuri și va trebui să o schimbi în tandem → DRY. Dacă soluția ta are mai mult de două niveluri de indirectare pentru o problemă care se poate exprima în 20 de linii → KISS.

05SOLID complet — cum colaborează cele 5 principii

Acum că am parcurs toate cinci principii, putem vedea cum se susțin reciproc. Nu sunt reguli independente — formează un sistem:

S
SRP — fiecare clasă are un singur motiv de schimbare
Produce clase mici și coezive. Face posibilă aplicarea celorlalte principii: o clasă cu o singură responsabilitate este mai ușor de „închis" pentru modificare (OCP) și mai ușor de testat în izolare (DIP).
O
OCP — deschis pentru extensie, închis pentru modificare
Se bazează pe polimorfism și interfețe. Funcționează corect numai dacă LSP este respectat: extensiile trebuie să fie substituibile. DIP face OCP fezabil la scara modulelor întregi.
L
LSP — subtipurile sunt substituibile fără surprize
Garantează că polimorfismul pe care OCP îl cere funcționează corect. ISP ajută LSP: interfețele mici sunt mai ușor de implementat fără NotSupportedException.
I
ISP — clienții depind doar de metodele pe care le folosesc
Produce interfețe precise. Ajută LSP (implementările nu sunt forțate să suporte metode absurde). Ajută DIP: interfețele mici sunt mai ușor de mockat în teste.
D
DIP — modulele de nivel înalt depind de abstracții, nu de implementări
Face testabilitatea posibilă (injectezi mock-uri). Face OCP posibil la nivel de module întregi: poți schimba întreaga implementare a bazei de date fără să atingi logica de business.
SRP spune ce trebuie să conțină o clasă. OCP spune cum trebuie să evolueze. LSP spune ce condiții trebuie să respecte ierarhiile. ISP spune cât de mari trebuie să fie interfețele. DIP spune în ce direcție trebuie să curgă dependențele. Toate cinci împreună produc cod care crește fără să se destrame.
