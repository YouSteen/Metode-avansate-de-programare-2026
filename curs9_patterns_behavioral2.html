Design Patterns Comportamentale — Partea II

Patru pattern-uri comportamentale care decuplează cum se desfășoară procesul: Template Method — schelet de algoritm cu pași variabili în subclase; Chain of Responsibility — lanț de handler-i care procesează cererea pe rând; State — comportament dependent de stare cu tranziții explicite; Mediator — coordonare centralizată între obiecte care altfel ar interacționa direct.

Obiective
  • Aplicați Template Method pentru algoritmi cu structură fixă și pași variabili; înțelegeți compromisurile moștenire vs. compoziție față de Strategy
  • Construiți pipeline-uri de procesare cu Chain of Responsibility; recunoașteți pattern-ul în ASP.NET Core Middleware
  • Eliminați lanțurile de if/switch pe enum cu State Pattern, gestionând tranzițiile invalide cu excepții explicite
  • Identificați rolul Mediator în decuplarea unei rețele de obiecte și legătura cu MediatR / CQRS

01Context — continuarea pattern-urilor comportamentale

În cursul 8 am studiat patru pattern-uri comportamentale care decuplează cine vorbește cu cine: Observer (publish/subscribe), Strategy (algoritm interschimbabil), Command (operația ca obiect) și Iterator (parcurgere uniformă). Tema centrală era decuplarea responsabilităților între obiecte.

Partea II abordează o întrebare diferită: cum decuplăm modul în care se desfășoară un proces? Cum încapsulăm un algoritm cu pași variabili (Template Method)? Cum construim un lanț de procesare extensibil (Chain of Responsibility)? Cum modelăm un obiect care își schimbă comportamentul în funcție de starea internă (State)? Cum coordonăm o rețea de obiecte fără a crea legături directe (Mediator)?

02Template Method

📐
Template Method
Definește scheletul unui algoritm într-o metodă, lăsând unii pași subclaselor. Subclasele pot redefini anumiți pași fără a schimba structura algoritmului.
Comportamental GoF

Template Method codifică o regulă pe care o întâlnim frecvent: algoritmul este același peste tot, dar câțiva pași variază. Procesul de generare a unui raport e mereu același — încarci date, le filtrezi, le formatezi, le exportezi — dar fiecare tip de raport încarcă din altă sursă și formatează diferit. În loc să copiezi structura algoritmului în fiecare subclasă (cu riscul ca pașii să se dezsincronizeze), o definești o singură dată în clasa de bază și lași subclasele să umple golurile.

Structura C# — schelet de algoritm

Template Method — structura canonică C# 
public abstract class DataExporter
{
    // Template method — sealed pentru ca subclasele să NU îl rescrie
    public string Export()
    {
        var data       = LoadData();          // pas abstract
        var filtered   = FilterData(data);    // pas cu default
        var formatted  = FormatData(filtered); // pas abstract
        OnBeforeWrite(formatted);              // hook opțional
        return WriteOutput(formatted);          // pas abstract
    }

    // Pași obligatorii — fiecare subclasă TREBUIE să-i implementeze
    protected abstract IEnumerable<Record> LoadData();
    protected abstract string FormatData(IEnumerable<Record> data);
    protected abstract string WriteOutput(string content);

    // Pas cu default — subclasele POT să-l rescrie
    protected virtual IEnumerable<Record> FilterData(IEnumerable<Record> data)
        => data.Where(r => !r.IsDeleted); // implicit: exclude șterse

    // Hook — gol implicit; subclasele decid dacă suprascriu
    protected virtual void OnBeforeWrite(string content) { }
}

Trei tipuri de pași într-un template method:

  • abstractobligatoriu — subclasa trebuie să-l implementeze, altfel codul nu compilează
  • virtualare implementare implicită; subclasa poate să o rescrie
  • hookmetodă virtuală goală; punct de extensie opțional — implicit nu face nimic
Exemplul 1 Rapoarte de business — același schelet, surse diferite

Aplicația trebuie să exporte trei tipuri de rapoarte: vânzări (din baza de date), inventar (din ERP-ul extern), audit (din log-uri). Algoritmul de export este identic; doar sursa și formatul diferă.

ReportProcessor.cs C# 
public abstract class ReportProcessor
{
    public string GenerateReport(DateRange period)
    {
        Console.WriteLine($"[{GetType().Name}] Început raport pentru {period}");
        var data      = LoadData(period);
        var filtered  = FilterData(data);
        var formatted = FormatData(filtered);
        var path      = WriteToFile(formatted);
        OnReportGenerated(path); // hook
        return path;
    }

    protected abstract IEnumerable<Record> LoadData(DateRange period);
    protected abstract string FormatData(IEnumerable<Record> data);

    protected virtual IEnumerable<Record> FilterData(IEnumerable<Record> data)
        => data.Where(r => r.IsValid);

    protected virtual string WriteToFile(string content)
    {
        var path = Path.Combine(Path.GetTempPath(), $"report-{Guid.NewGuid():N}.txt");
        File.WriteAllText(path, content);
        return path;
    }

    protected virtual void OnReportGenerated(string path) { } // hook gol
}

// ConcreteClass 1 — raport vânzări din SQL
public class SalesReportProcessor : ReportProcessor
{
    private readonly ISalesRepository _repo;
    public SalesReportProcessor(ISalesRepository repo) => _repo = repo;

    protected override IEnumerable<Record> LoadData(DateRange p)
        => _repo.QuerySales(p.Start, p.End).Select(s => new Record(s));

    protected override string FormatData(IEnumerable<Record> data)
        => "=== VÂNZĂRI ===\n" + string.Join("\n", data.Select(d => d.AsRow()));

    // Hook suprascris — trimitem email când raportul e gata
    protected override void OnReportGenerated(string path)
        => Console.WriteLine($"📧 Trimit email cu raport către sales@...");
}

// ConcreteClass 2 — raport audit din log-uri; nu filtrăm (toate intrările contează)
public class AuditReportProcessor : ReportProcessor
{
    protected override IEnumerable<Record> LoadData(DateRange p)
        => File.ReadAllLines("audit.log").Select(line => Record.Parse(line));

    protected override string FormatData(IEnumerable<Record> data)
        => JsonSerializer.Serialize(data, new JsonSerializerOptions { WriteIndented = true });

    // Audit-ul include TOATE intrările — suprascriem filtrul implicit
    protected override IEnumerable<Record> FilterData(IEnumerable<Record> data) => data;
}

var sales = new SalesReportProcessor(repo);
sales.GenerateReport(new DateRange(DateTime.Today.AddDays(-30), DateTime.Today));
// [SalesReportProcessor] Început raport pentru ...
// 📧 Trimit email cu raport către sales@...
💡
Principiul Hollywood

Template Method ilustrează „Hollywood Principle": nu ne suna noi, te sunăm noi. Clientul apelează GenerateReport() pe clasa de bază; clasa de bază apelează metodele subclasei. Inversiunea fluxului de control este miezul pattern-ului — subclasa nu definește algoritmul, doar îl completează.

Template Method vs. Strategy

Ambele rezolvă o problemă similară — variabilitatea unui algoritm — dar prin mecanisme fundamental diferite. Template Method folosește moștenire; Strategy folosește compoziție. Alegerea greșită produce cod fragil.

Template Method — moștenire
public abstract class PaymentProcessor
{
    public void Process(Order o)
    {
        Validate(o);
        Charge(o);     // abstract
        Confirm(o);
    }
    protected abstract void Charge(Order o);
}

public class CardPayment : PaymentProcessor
{
    protected override void Charge(Order o)
        => stripe.Charge(o);
}

// Algoritmul e fix; doar Charge variază.
// Schimbarea = clasă nouă derivată.
Strategy — compoziție
public class PaymentProcessor
{
    private readonly IChargeStrategy _strategy;

    public PaymentProcessor(IChargeStrategy s)
        => _strategy = s;

    public void Process(Order o)
    {
        Validate(o);
        _strategy.Charge(o); // delegat
        Confirm(o);
    }
}

// Strategia poate fi schimbată la runtime.
// Testabil cu mock.
Regula practică: dacă structura algoritmului este stabilă și doar pașii variază, Template Method e potrivit. Dacă alegerea algoritmului trebuie să fie dinamică sau testabilă în izolare, preferă Strategy. În C# modern, Strategy câștigă mai des — moștenirea fixează relații care injecția de dependențe le ține flexibile.
✓ Când se potrivește
Algoritmul are o structură fixă pe care vrei să o protejezi de modificări accidentale. Mai multe subclase duplică același flux, doar cu pași diferiți. Cerințele lasă loc subclaselor să extindă comportamentul, dar nu să rearanjeze pașii. Framework-uri și clase de bază reutilizabile (ASP.NET Controller, EF DbContext.OnModelCreating).
✗ Când nu se potrivește
Vrei să schimbi algoritmul la runtime (alegere dinamică) — Strategy e mai potrivit. Ierarhia de moștenire ar deveni profundă sau ar trebui să combini comportamente — compoziția scapă din această capcană. Există o singură subclasă reală — moștenirea adaugă ceremonie fără beneficii.

03Chain of Responsibility

🔗
Chain of Responsibility
Evită cuplarea sender-ului cu receiver-ul, dând mai multor obiecte șansa de a procesa cererea. Obiectele sunt înlănțuite; fiecare decide dacă procesează sau pasează cererea mai departe.
Comportamental GoF

Imaginează-ți validarea unei comenzi: trebuie să verifici stocul, prețurile, frauda, vârsta clientului. Soluția naivă — o singură metodă cu cinci if-uri în șir — funcționează, dar e rigidă: ordinea e fixă, adăugarea unei verificări noi cere modificarea metodei, iar dezactivarea condiționată a unei verificări (ex. în staging) cere flag-uri suplimentare. Chain of Responsibility împarte responsabilitatea: fiecare verificare devine un handler independent, iar lanțul lor se construiește explicit, ca o conductă prin care trece cererea.

Structura C# — handler de bază reutilizabil

Chain of Responsibility — handler bază C# 
// Interfața — toți handler-ii respectă același contract
public interface IHandler<TRequest, TResult>
{
    IHandler<TRequest, TResult> SetNext(IHandler<TRequest, TResult> next);
    TResult Handle(TRequest request);
}

// Bază abstractă — captează tot codul comun (next, fluent SetNext)
public abstract class BaseHandler<TRequest, TResult> : IHandler<TRequest, TResult>
{
    protected IHandler<TRequest, TResult>? Next { get; private set; }

    // Returnează handler-ul primit pentru a permite înlănțuire fluent
    public IHandler<TRequest, TResult> SetNext(IHandler<TRequest, TResult> next)
    {
        Next = next;
        return next;
    }

    public abstract TResult Handle(TRequest request);
}

Două variante fundamentale

Chain of Responsibility apare în două variante esențial diferite, după ce face un handler care a procesat cererea:

Varianta A Short-circuit — primul handler care răspunde oprește lanțul

Tipic pentru validare sau autorizare: prima problemă găsită determină răspunsul; nu mai are sens să continui. Folosit și pentru routing — primul handler care recunoaște cererea o servește.

OrderValidationChain.cs — short-circuit la primul eșec C# 
public record ValidationResult(bool IsValid, string? Error);

public abstract class OrderValidator : BaseHandler<Order, ValidationResult>
{
    public override ValidationResult Handle(Order order)
    {
        var localResult = Validate(order);
        if (!localResult.IsValid) return localResult; // STOP — prima eroare
        return Next?.Handle(order) ?? new(true, null);
    }
    protected abstract ValidationResult Validate(Order order);
}

public class StockValidator : OrderValidator
{
    protected override ValidationResult Validate(Order o)
        => o.Items.All(i => i.InStock)
            ? new(true, null)
            : new(false, "Stoc insuficient pentru unul sau mai multe produse");
}

public class PriceValidator : OrderValidator
{
    protected override ValidationResult Validate(Order o)
        => o.Total > 0
            ? new(true, null)
            : new(false, "Totalul comenzii trebuie să fie pozitiv");
}

public class FraudValidator : OrderValidator
{
    protected override ValidationResult Validate(Order o)
        => o.Total < 10_000m || o.Customer.IsTrusted
            ? new(true, null)
            : new(false, "Comandă suspectă — necesită verificare manuală");
}

// Construirea lanțului — Builder fluent prin SetNext care returnează next
var chain = new StockValidator();
chain
    .SetNext(new PriceValidator())
    .SetNext(new FraudValidator());

var result = chain.Handle(order);
if (!result.IsValid)
    Console.WriteLine($"❌ {result.Error}");
// Dacă stoc OK, dar preț = 0 → "Totalul comenzii trebuie să fie pozitiv"
// FraudValidator nu mai e apelat — short-circuit.
Varianta B Pass-through — toți handler-ii rulează, fiecare contribuie

Tipic pentru pipeline-uri de transformare: fiecare handler modifică sau îmbogățește cererea, apoi o pasează la următorul. Lanțul rulează până la capăt indiferent ce face fiecare verigă. ASP.NET Core Middleware funcționează exact așa — fiecare middleware are șansa să modifice cererea înainte și răspunsul după.

LoggingPipeline.cs — pass-through C# 
public abstract class RequestEnricher : BaseHandler<HttpRequest, HttpRequest>
{
    public override HttpRequest Handle(HttpRequest request)
    {
        var enriched = Enrich(request); // fiecare handler contribuie
        return Next?.Handle(enriched) ?? enriched; // CONTINUĂ
    }
    protected abstract HttpRequest Enrich(HttpRequest request);
}

public class AddCorrelationId : RequestEnricher
{
    protected override HttpRequest Enrich(HttpRequest r)
        => r.With("X-Correlation-Id", Guid.NewGuid().ToString());
}

public class AddUserContext : RequestEnricher
{
    protected override HttpRequest Enrich(HttpRequest r)
        => r.With("X-User", GetCurrentUser().Id);
}

public class LogTimestamp : RequestEnricher
{
    protected override HttpRequest Enrich(HttpRequest r)
    {
        Console.WriteLine($"[{DateTime.UtcNow:O}] {r.Path}");
        return r; // nu modifică, doar observă
    }
}

var pipeline = new AddCorrelationId();
pipeline
    .SetNext(new AddUserContext())
    .SetNext(new LogTimestamp());

var finalRequest = pipeline.Handle(incoming);
// Toate cele trei rulează indiferent — request-ul a fost îmbogățit cu trei header-e
🧭
Chain of Responsibility în .NET — ASP.NET Core Middleware

Întreaga conductă HTTP din ASP.NET Core este Chain of Responsibility varianta pass-through. app.UseAuthentication(); app.UseAuthorization(); app.UseRouting(); construiește un lanț unde fiecare middleware decide să cheme sau nu await next(). Dacă apelează next() — pass-through; dacă răspunde direct fără să cheme next() — short-circuit. Pattern-ul nu cunoaște alegerea în avans; o face fiecare middleware concret.

✓ Când se potrivește
Pipeline-uri de validare, autorizare, logging, transformare. Lista de pași poate crește în timp și ordinea contează. Pași pot fi activați/dezactivați condiționat (configurare, mediu). Nu se știe în avans cine va răspunde — handler-ul potrivit decide singur.
✗ Când nu se potrivește
Există un destinatar fix și cunoscut — un apel direct e mai simplu. Toți pașii trebuie să ruleze obligatoriu indiferent — o secvență de metode într-o clasă e mai clară. Lanțul devine prea lung sau dependențele dintre handler-i sunt implicite — debugging-ul fluxului devine dificil.

04State

🔄
State
Permite unui obiect să-și schimbe comportamentul când starea sa internă se schimbă. Obiectul pare că își schimbă clasa. Tranzițiile între stări sunt explicite și gestionate de stările concrete.
Comportamental GoF

State Pattern rezolvă o problemă pe care toată lumea o întâlnește în obiecte cu ciclu de viață: o comandă trece prin Pending → Confirmed → Processing → Shipped → Delivered; un document prin Draft → Review → Published; un transfer bancar prin Initiated → Authorized → Settled. Soluția naivă — un enum și switch-uri în fiecare metodă — funcționează pentru două-trei stări, dar devine rapid neîntreținut.

Problema fără State

✗ OrderWithoutState.cs — un singur enum, switch peste tot C# 
public enum OrderStatus { Pending, Confirmed, Processing, Shipped, Delivered, Cancelled }

public class Order
{
    public OrderStatus Status { get; private set; } = OrderStatus.Pending;

    public void Pay()
    {
        switch (Status)
        {
            case OrderStatus.Pending:    Status = OrderStatus.Confirmed; break;
            case OrderStatus.Cancelled:  throw new InvalidOperationException("Anulată");
            default: throw new InvalidOperationException($"Nu pot plăti în starea {Status}");
        }
    }

    public void Ship()
    {
        switch (Status) { /* alt switch identic ca structură */ }
    }

    public void Cancel()
    {
        switch (Status) { /* iar un switch — replicarea cunoașterii despre stări */ }
    }

    // Problemă: o stare nouă (ex. Returning) cere modificarea fiecărui switch.
    // Tranzițiile valide sunt împrăștiate în întreaga clasă.
    // Adăugarea unei metode noi multiplică switch-urile.
}

Trei probleme clare apar din această abordare: cunoașterea despre tranziții este împrăștiată, fiecare schimbare cere modificări în multiple locuri (OCP violat), iar testarea cere acoperirea tuturor combinațiilor stare × metodă.

Exemplul 1 Order state machine — refactorizare completă

Trecem aceeași comandă printr-un model bazat pe State. Fiecare stare concretă știe exact ce tranziții permite și pe care le respinge — cunoașterea e localizată.

OrderState.cs C# 
// Interfața — toate tranzițiile permise în domeniu
public interface IOrderState
{
    string Name { get; }
    void Pay(Order order);
    void Process(Order order);
    void Ship(Order order);
    void Deliver(Order order);
    void Cancel(Order order);
}

// Excepție pentru tranziții invalide — clară, expresivă
public class InvalidOrderTransitionException : InvalidOperationException
{
    public InvalidOrderTransitionException(string from, string action)
        : base($"Nu pot executa '{action}' în starea '{from}'") { }
}

// Context — Order deleghează la starea curentă
public class Order
{
    internal IOrderState CurrentState { get; set; } = new PendingState();
    public string Status => CurrentState.Name;

    public void Pay()     => CurrentState.Pay(this);
    public void Process() => CurrentState.Process(this);
    public void Ship()    => CurrentState.Ship(this);
    public void Deliver() => CurrentState.Deliver(this);
    public void Cancel()  => CurrentState.Cancel(this);
}

// Starea Pending — comandă creată, neplătită
public class PendingState : IOrderState
{
    public string Name => "Pending";
    public void Pay(Order o)     => o.CurrentState = new ConfirmedState();
    public void Cancel(Order o)  => o.CurrentState = new CancelledState();
    public void Process(Order o) => throw new InvalidOrderTransitionException(Name, "Process");
    public void Ship(Order o)    => throw new InvalidOrderTransitionException(Name, "Ship");
    public void Deliver(Order o) => throw new InvalidOrderTransitionException(Name, "Deliver");
}

public class ConfirmedState : IOrderState
{
    public string Name => "Confirmed";
    public void Process(Order o) => o.CurrentState = new ProcessingState();
    public void Cancel(Order o)  => o.CurrentState = new CancelledState();
    public void Pay(Order o)     => throw new InvalidOrderTransitionException(Name, "Pay");
    public void Ship(Order o)    => throw new InvalidOrderTransitionException(Name, "Ship");
    public void Deliver(Order o) => throw new InvalidOrderTransitionException(Name, "Deliver");
}

public class ShippedState : IOrderState
{
    public string Name => "Shipped";
    public void Deliver(Order o) => o.CurrentState = new DeliveredState();
    // Atenție: Shipped NU mai poate fi anulat — Cancel aruncă excepție
    public void Cancel(Order o)  => throw new InvalidOrderTransitionException(Name, "Cancel");
    public void Pay(Order o)     => throw new InvalidOrderTransitionException(Name, "Pay");
    public void Process(Order o) => throw new InvalidOrderTransitionException(Name, "Process");
    public void Ship(Order o)    => throw new InvalidOrderTransitionException(Name, "Ship");
}

// Stările terminale — nicio tranziție
public class DeliveredState : IOrderState { /* toate metodele aruncă */ }
public class CancelledState : IOrderState { /* toate metodele aruncă */ }

var order = new Order();
Console.WriteLine(order.Status); // Pending
order.Pay();                       // → Confirmed
order.Process();                   // → Processing
order.Ship();                      // → Shipped
order.Cancel();                    // 💥 InvalidOrderTransitionException
✓
Câștigul concret

O stare nouă (ex. Returning) = o singură clasă nouă. Cunoașterea despre ce e permis trăiește într-un singur loc per stare. Testele unitare devin granulare: testezi PendingState.Pay() izolat, nu un switch de 100 de linii cu zeci de combinații.

Diagrama de stări — vizualizare obligatorie

Înainte de a scrie cod cu State Pattern, desenează diagrama de stări. Te forțează să identifici stările terminale, să răspunzi explicit ce se întâmplă pentru fiecare combinație stare × acțiune și să descoperi tranziții lipsă.

Cine decide tranzițiile?

Există două abordări de a încredința tranzițiile:

  • decentralizatStarea concretă decide tranziția — PendingState.Pay() setează singură CurrentState = new ConfirmedState(). Avantaj: cunoașterea e localizată. Dezavantaj: stările trebuie să cunoască celelalte stări (cuplare slabă, dar prezentă).
  • centralizatContext-ul decide tranziția pe baza unui dicționar sau a unei state-table. Stările conțin doar comportament; tabela definește grafu de tranziții. Avantaj: graful e vizibil într-un singur loc. Dezavantaj: mai puțin „OO", mai mult „data-driven".

Pentru un număr mic de stări (5-10) și tranziții relativ simple, varianta descentralizată este mai naturală în C# și ușor de citit. Pentru sisteme cu zeci de stări sau workflow-uri configurabile, considerați biblioteca Stateless — o bibliotecă .NET dedicată state machines.

State vs. Strategy

Ambele pattern-uri folosesc compoziție pentru a delega comportament unui obiect intercambiabil. Diferența nu este structurală, ci semantică:

AspectStrategyState
Cine alege obiectul?Clientul, în mod explicitObiectul însuși, prin tranziții
Cunoaște alte instanțe similare?Nu — strategiile sunt independenteDa — stările trec una în alta
Există ordine între ele?Nu — pot fi alese în orice secvențăDa — graf de tranziții
Schimbare la runtime?La cererea clientuluiCa efect al unei acțiuni interne
Exemplu tipicAlgoritm de sortare, calcul preț, format exportWorkflow, life cycle, protocol
✓ Când se potrivește
Obiectul are un ciclu de viață cu reguli clare despre ce e permis în fiecare stare. Soluția actuală e plină de switch-uri pe enum status. Tranzițiile invalide trebuie semnalate explicit, nu ignorate tăcut. Workflow-uri, protocoale, automate finite.
✗ Când nu se potrivește
Există doar 2 stări binare și o singură tranziție (on/off). Diferențele între stări sunt doar de date, nu de comportament — un câmp simplu e suficient. Tranzițiile sunt arbitrare și foarte numeroase — o bibliotecă specializată (Stateless) e mai potrivită.

05Mediator

🎭
Mediator
Definește un obiect care încapsulează modul în care un set de obiecte interacționează. Promovează cuplarea slabă prin împiedicarea obiectelor să se refere explicit unele la altele, dându-le posibilitatea să-și varieze interacțiunile independent.
Comportamental GoF Conceptual

Problema rezolvată de Mediator: într-un sistem cu N componente care comunică direct între ele, numărul de legături potențiale este N×(N-1)/2 — graf complet, dezastruos pentru întreținere. Mediator înlocuiește acest graf N×N cu o stea N×1 — toate componentele vorbesc doar cu mediatorul, iar mediatorul coordonează cine vorbește cu cine.

Structura

Diagramă de clase

ConcreteMediator Component <|-- ComponentA Component <|-- ComponentB ConcreteMediator o--> ComponentA ConcreteMediator o--> ComponentB Component --> Mediator
Exemplul 1 Formular cu controale interdependente

Un formular de înregistrare are: un dropdown Country, un text-box City și un checkbox SubscribeNewsletter. Reguli: schimbarea țării resetează orașul; bifarea newsletter cere obligatoriu un email valid. Fără Mediator, fiecare control trebuie să cunoască celelalte. Cu Mediator, controalele anunță doar evenimentul; mediatorul aplică regulile.

RegistrationFormMediator.cs C# 
public interface IFormMediator
{
    void Notify(object sender, string @event);
}

public class RegistrationFormMediator : IFormMediator
{
    public CountryDropdown   Country  { get; set; } = null!;
    public CityTextBox       City     { get; set; } = null!;
    public NewsletterCheckbox News    { get; set; } = null!;
    public EmailTextBox      Email    { get; set; } = null!;

    public void Notify(object sender, string @event)
    {
        switch ((sender, @event))
        {
            case (CountryDropdown, "changed"):
                City.Clear();
                City.LoadCitiesFor(Country.Value);
                break;

            case (NewsletterCheckbox, "checked"):
                Email.IsRequired = true;
                Email.Focus();
                break;

            case (NewsletterCheckbox, "unchecked"):
                Email.IsRequired = false;
                break;
        }
    }
}

// Componentele NU se cunosc între ele — știu doar de mediator
public abstract class FormControl
{
    protected readonly IFormMediator _mediator;
    protected FormControl(IFormMediator m) => _mediator = m;
}

public class CountryDropdown : FormControl
{
    private string _value = "";
    public string Value
    {
        get => _value;
        set { _value = value; _mediator.Notify(this, "changed"); }
    }
    public CountryDropdown(IFormMediator m) : base(m) { }
}

Componentele CountryDropdown și CityTextBox nu au nicio referință una la alta. Toată logica de coordonare e în mediator — adăugarea unei reguli noi (ex. „dacă newsletter e bifat, ascunde câmpul de telefon") schimbă doar mediatorul.

Mediator în .NET — MediatR și CQRS

În ecosistemul .NET, biblioteca MediatR este implementarea cea mai populară a Mediator, la nivel de aplicație. Acolo unde Mediator clasic conectează controale UI, MediatR conectează cereri (Commands și Queries) cu handler-e:

MediatR — Command + Handler C# 
// Request — comandă (cerere de scriere)
public record CreateOrderCommand(Guid CustomerId, List<OrderItem> Items) : IRequest<Guid>;

// Handler — răspunde la cererea de tip specific
public class CreateOrderHandler : IRequestHandler<CreateOrderCommand, Guid>
{
    private readonly IOrderRepository _repo;
    public CreateOrderHandler(IOrderRepository repo) => _repo = repo;

    public async Task<Guid> Handle(CreateOrderCommand cmd, CancellationToken ct)
    {
        var order = Order.Create(cmd.CustomerId, cmd.Items);
        await _repo.SaveAsync(order, ct);
        return order.Id;
    }
}

// Controller-ul nu cunoaște handler-ul — doar mediatorul
public class OrdersController : ControllerBase
{
    private readonly IMediator _mediator;
    public OrdersController(IMediator m) => _mediator = m;

    [HttpPost]
    public async Task<IActionResult> Create([FromBody] CreateOrderCommand cmd)
    {
        var id = await _mediator.Send(cmd); // mediatorul rutează la handler
        return CreatedAtAction(nameof(Get), new { id }, id);
    }
}

Pregătire pentru cursul 12

MediatR este coloana vertebrală a multor implementări de CQRS (Command Query Responsibility Segregation) în .NET — pattern arhitectural pe care îl vom studia în cursul 12. Atunci veți vedea cum aceeași idee de mediator se extinde la separarea operațiilor de scriere (Commands) de cele de citire (Queries) și la integrarea cu evenimente de domeniu.

✓ Când se potrivește
Rețea de obiecte care interacționează în multiple moduri (UI controls, microservicii intra-process). Logica de coordonare merită propriul obiect, separat de componente. Vrei să adaugi/elimini participanți fără a modifica ceilalți. Integrare cu CQRS / Clean Architecture (handler-e per request).
✗ Când nu se potrivește
Doar 2-3 obiecte care interacționează — un apel direct e mai clar. Mediatorul ar deveni un God Object cu logica coordonării prea complexă. Componentele oricum trebuie să cunoască multă structură comună — Mediator nu mai aduce decuplare.

06Comparație — cele 4 pattern-uri de astăzi

PatternProblema rezolvatăMecanism principalAlternativă modernă
Template Method Algoritm cu structură fixă, pași variabili moștenire + metode abstracte Strategy + compoziție
Chain of Resp. Lanț de pași de procesare linked list de handler-i ASP.NET Core Middleware
State Obiect cu comportament dependent de stare delegare la state object biblioteca Stateless
Mediator Coordonare între obiecte interconectate obiect central de mesaje MediatR (CQRS)
Dacă pattern-urile din Partea I (Observer/Strategy/Command/Iterator) au decuplat cine vorbește cu cine, cele din Partea II decuplează cum se desfășoară procesul: Template Method protejează scheletul unui algoritm; Chain of Responsibility transformă un lanț de if-uri într-o conductă extensibilă; State elimină switch-urile pe enum; Mediator înlocuiește graful N×N cu o stea.

Decision tree — ce pattern aleg?

  • Ai un switch / if-else lanț pe un enum status? → State Pattern
  • Ai un lanț de verificări / validări / transformări? → Chain of Responsibility
  • Ai mai multe clase cu același flux dar pași diferiți? → Template Method (sau Strategy)
  • Ai o rețea de obiecte care comunică direct între ele? → Mediator

Mapping pattern → rol în aplicație

PatternRol concretRealizare practică
Chain of Resp. Validează cererea de creare comandă; primul handler care detectează problemă oprește chain-ul IOrderValidationHandler + 4 handler-i concreți, short-circuit la primul eșec

Cerințe
DOMAIN Cerința 1 — Modelul de domeniu
  • Order — Aggregate Root, conține lista de items, customer, adresa de livrare, total calculat, starea curentă
  • OrderItem — record cu ProductId, ProductName, Quantity, UnitPrice, HasAgeRestriction
  • Customer — record cu Id, Name, Email, Age, IsTrusted
  • Value Objects: Money (sumă + valută), Address (stradă, oraș, codPoștal, țară), OrderId (wrap peste Guid)
  • Order.CurrentState este expus prin proprietate read-only spre exterior; doar metodele de tranziție (Pay, Process, Ship, Deliver, Cancel) îl pot schimba
  • Order.History — listă de tranziții (string FromState, string ToState, DateTime At) populată automat la fiecare schimbare
CHAIN Cerința 2 — Chain of Responsibility: pipeline de validare
  • Interfața IOrderValidationHandler cu SetNext(handler) și Handle(ValidationContext); returnează ValidationResult
  • BaseValidationHandler — clasă abstractă cu metoda Handle implementată conform variantei short-circuit (oprește la primul eșec) și metoda abstractă Validate(context) pe care subclasele o implementează
  • StockValidationHandler — verifică că toate produsele din comandă sunt în stoc; folosește un repository in-memory de produse (suficient un Dictionary<Guid, int> hardcodat)
  • PriceValidationHandler — verifică că prețurile sunt pozitive și că totalul calculat coincide cu suma Quantity × UnitPrice pentru fiecare item
  • FraudDetectionHandler — refuză comenzile peste 10.000 RON dacă Customer.IsTrusted == false; refuză și mai mult de 50 de items distincte într-o singură comandă
  • AgeVerificationHandler — dacă vreun OrderItem.HasAgeRestriction == true, customer-ul trebuie să aibă Age >= 18
  • Chain-ul se construiește prin SetNext() fluent (vezi exemplul din curs); ordinea: Stock → Price → Fraud → Age
  • ValidationResult este un record cu IsValid (bool) și Errors (listă) — pe short-circuit returnează un singur error, restul handler-ilor nu rulează
STATE Cerința 3 — State: ciclul de viață al comenzii
  • Interfața IOrderState cu metodele: Pay(order), Process(order), Ship(order), Deliver(order), Cancel(order) + proprietate Name
  • 6 stări concrete: PendingState (initial), ConfirmedState, ProcessingState, ShippedState, DeliveredState (terminal), CancelledState (terminal alternativ)
  • Tranziții permise (decentralizate — fiecare stare decide):
    • Pending → Confirmed prin Pay(); → Cancelled prin Cancel()
    • Confirmed → Processing prin Process(); → Cancelled prin Cancel()
    • Processing → Shipped prin Ship(); → Cancelled prin Cancel()
    • Shipped → Delivered prin Deliver(); Cancel() NU este permis după Shipped
    • Delivered și Cancelled — terminale, nicio tranziție nu este permisă
  • Tranzițiile invalide aruncă InvalidOrderTransitionException cu mesaj clar despre starea curentă și acțiunea încercată
  • Order deleghează toate metodele către CurrentState; NU conține switch pe starea curentă — toată cunoașterea e în stările concrete
  • Fiecare tranziție reușită adaugă o intrare în Order.History
SERVICE Cerința 4 — OrderService: punct de intrare
  • OrderService este orchestratorul; primește prin DI: IOrderRepository (in-memory) și chain-ul de validare (construit la pornire)
  • Metodă CreateOrder(CreateOrderRequest): rulează chain-ul de validare; dacă valid, creează Order și o salvează; returnează Order sau erorile de validare
  • Metode pentru tranziții: PayOrder(id), ProcessOrder(id), ShipOrder(id), DeliverOrder(id), CancelOrder(id) — fiecare apelează metoda corespunzătoare pe Order și salvează modificarea
  • Metode de query: GetOrder(id) returnează Order sau null; GetAllOrders() returnează lista completă
  • Excepțiile InvalidOrderTransitionException NU sunt prinse aici — trec până la nivel de endpoint, unde se convertesc în răspuns HTTP 409

  2. Chain of Responsibility cu short-circuit (oprire la primul răspuns) vs. pass-through (toți handler-ii rulează) — cum decideți care variantă se potrivește problemei? Există cazuri în care vă trebuie un comportament hibrid?
  3. State Pattern descentralizat (starea decide tranziția) vs. centralizat (context-ul are tabelul de tranziții). Pentru o aplicație cu 20+ stări și workflow configurabil de admin, care abordare alegeți și de ce?
  4. Mediator poate deveni un God Object dacă concentrează prea multă logică de coordonare. Care sunt semnele că ați depășit zona sănătoasă și ar trebui să descompuneți mediatorul în mai mulți mediatori specializați?
